# コードレビュー

## tagline

書いたコードを他者の目で確認し、品質・設計・安全性を高める工程です。

<!-- ━━━━━━━━ 左ページ ━━━━━━━━ -->

## 何をしてくれるか

PR（プルリクエスト）やコードリーディング会を通じて、機能の正しさ・設計の一貫性・可読性・セキュリティなどの観点で指摘と提案を行います。書き手が気づきにくい問題を早期に発見できます。


## どこで出会うか

GitHub 上の PR（F-61）に「コメントをつける」形で日常的に登場します。AI ツールが PR の内容を自動で粗取りし、人間レビュアーに設計議論を集中させる運用が広まっています。


## メイン図

### 図の狙い

「AI による事前レビュー → 人間による設計レビュー」という二段構えの流れを示し、役割分担を掴んでもらう。

### C. 概念図（figure_type: workflow）

- 中心に置く概念: レビュー工程（AI 粗取り → 人間確認）
- 周辺の要素: PR / ESLint / CodeRabbit / 設計観点 / マージ判断
- 関係の描き方: 左から右への矢印フロー


## 会話での使い方例

「Claude に PR の事前レビューをさせると、人間が見るときには設計の話に集中できます。」

## この用語の見どころ

### 1. 役割

他者の目でコードの問題を洗い出す品質管理の関門です。

### 2. うれしさ

バグや設計ミスを本番到達前に発見でき、手戻りが減ります。

### 3. 注意点

規約の細かい指摘は機械（Linter）に任せ、人間は設計に集中します。

### 4. どこで役立つか

チーム開発や AI 生成コードの確認工程で効果が出やすいです。

### 5. はじめに

PR へのコメントと Linter 自動チェックの役割分担を把握します。

### 6. 深掘り先

Pull Request、ESLint、CodeRabbit


## 開発フローでの位置（必須）

1. コード実装 — 機能を書いてコミットします
2. PR 作成 — Pull Request を開き、差分を共有します
3. AI 事前レビュー — CodeRabbit などが自動で粗取りします
4. 人間レビュー — 設計・意図・安全性を確認してマージ判断します

## 誌面ポンチ絵メモ

### メイン図（左ページ中段 / figure_type: workflow）

- 描く内容: AI と人間が二段でコードを確認するフロー図
- 登場人物: 開発者（コードを書く人）とレビュアー（人間）、AI ロボットアイコン

- 中央に置くキーワード/ラベル: コードレビュー（二段構え）
- Before / After の場合の対比ポイント: Before = 人間が細かい指摘に時間を取られる / After = AI が粗取り、人間は設計議論に集中

## コミュニティ補完メモ

- H-2（ペアプログラミング）との住み分け：H-2 は同期・リアルタイムの共同作業。H-4 は非同期の PR ベースレビューが主軸。重複する場合はシーンで使い分ける。
- F-61（Pull Request）との住み分け：F-61 は変更を取り込む仕組みそのもの。H-4 はその工程の中で行う「確認・指摘」の行為に焦点。
- F-20（ESLint）／F-21（Prettier）との住み分け：機械的な規約チェックは Linter 側のエントリに任せる。H-4 では「Linter に任せて人間は設計議論に集中する」という方針の説明にとどめる。
