# ベクトル DB

## tagline

意味の近さで検索する専用データベースです。Embedding（ベクトル）を保存し、似た内容を高速に返します。

<!-- ━━━━━━━━ 左ページ ━━━━━━━━ -->

## 何をしてくれるか

数百万件以上のベクトルを保存し、入力と意味的に近い上位 N 件をミリ秒単位で返します。近似最近傍探索（HNSW）で全件比較せずに高速検索を実現します。


## どこで出会うか

RAG（G-15）の実装記事や社内検索の構成図でよく登場します。Pinecone・Chroma・Qdrant などの製品名や、PostgreSQL（F-120）の拡張 pgvector として既存 DB に組み込む例で見かけます。


## メイン図

### 図の狙い

「普通の DB はキー一致で返す、ベクトル DB は距離で返す」という検索軸の違いを 1 枚で示します。

### C. 概念図（figure_type: structure）

- 中心に置く概念: ベクトル空間（高次元）
- 周辺の要素: 質問ベクトル（入力） / 保存済みベクトル群 / 上位 N 件の返却 / コサイン距離ラベル
- 関係の描き方: 点の集合と「最近傍」への矢印


## 会話での使い方例

「最初は pgvector で十分でしたが、数千万件を超えたら Pinecone に乗り換えました。」

## この用語の見どころ

### 1. 役割

Embedding を格納し、意味の近いものを高速に取り出す検索エンジンです。

### 2. うれしさ

全件比較なしで大量データから類似結果を返せる点が助かります。

### 3. 注意点

件数が少ない段階では普通の DB で十分なことがあります。

### 4. どこで役立つか

RAG の検索インデックスや類似商品・重複検出の場面で役立ちます。

### 5. はじめに

「RDB は値一致、ベクトル DB は距離順」という違いを押さえると入門できます。

### 6. 深掘り先

RAG、Embedding、HNSW


## 開発フローでの位置（必須）

1. テキスト収集 — ドキュメントや FAQ など検索対象のテキストを用意します
2. Embedding 生成 — 各テキストをモデルでベクトル化してインデックスに登録します
3. クエリ変換 — ユーザーの入力も同じモデルでベクトル化します
4. 類似検索 — ベクトル DB に問い合わせて上位 N 件を取得し、LLM に渡します

## コミュニティ補完メモ

- Embedding（G-16）との住み分け：G-16 は「ベクトルの作り方」を扱い、G-17 はそのベクトルを「保存・検索する入れ物」を扱います。セットで読む設計です。
- RAG（G-15）との住み分け：G-15 は RAG 全体の構成を扱い、G-17 はその中の検索インデックス部分にフォーカスします。

## 備考

- 件数の目安：pgvector は数百万件規模まで十分な速度が出る事例が多く、それを超える場合に専用 SaaS（Pinecone 等）や OSS（Milvus / Qdrant）への移行が検討されます。
- 料金：Pinecone はスターターが無料枠あり（時変情報のため evaluation_date 確認推奨）。Chroma と Qdrant はローカル無料で起動可能。
- 「ベクトル DB」と「ベクトルストア」は同義で使われることがあります。
