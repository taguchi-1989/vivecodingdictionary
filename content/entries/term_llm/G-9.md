# effort レベル

<!-- ━━━━━━━━ 左ページ ━━━━━━━━ -->

## tagline

Thinking（シンキング）モデルに「どれだけ深く考えるか」を指示する設定です。low／medium／high の 3 段階が一般的です。

## 何をしてくれるか

モデル内部の推論ステップ数を呼び出し側が制御できます。high にすると思考時間が長くなり正答率が上がる反面、料金とレイテンシが増えます。軽い作業には low を使い、コストを抑えることができます。

## どこで出会うか

Claude Code では `--effort high` フラグや settings.json で指定する場面が代表的です。OpenAI API では `reasoning_effort` パラメータとして登場します。「複雑なバグ調査は high、コメント修正は low」のような使い分けが現場で広がっています。

## メイン図

### 図の狙い

low／medium／high の 3 段階を横に並べ、思考時間・料金・精度の変化を一目で示します。

### B. 登場シーン（figure_type: comparison）

- シーン1: 開発者が Claude Code で `--effort high` を指定して難解なバグを調査する
- シーン2: OpenAI API で `reasoning_effort: low` を渡して高速な整形タスクをこなす
- シーン3: settings.json にデフォルト値を書いておき、常に medium で動かす
- 並べる基準: effort 値が変わると何が変わるかを用途別に対比

## 会話での使い方例

「このバグ調査は effort を high にしないと Thinking モデルが浅い答えを返すことがあります。」

## この用語の見どころ

### 1. 役割

Thinking モデルの内部推論量をパラメータで制御します。

### 2. うれしさ

タスクの難度に合わせてコストと精度のバランスを取れます。

### 3. 注意点

high でも正答を保証するものではなく、料金が増える点に注意が必要です。

### 4. どこで役立つか

複雑なバグ調査や設計レビューで high を選ぶと効果的です。

### 5. はじめに

low／medium／high の意味と、それぞれの料金差を把握することが出発点です。

### 6. 深掘り先

Thinking モデル、reasoning_effort、o3

## 開発フローでの位置（必須）

1. タスク難度の見極め — 「整形だけか、深い推論が要るか」を判断します
2. effort 値の選択 — low／medium／high から 1 つを選びます
3. API またはフラグで指定 — `--effort high` や `reasoning_effort` パラメータに渡します
4. 結果と料金を確認 — 出力の質とトークン消費を照合します
5. デフォルト値の固定 — settings.json に書いて毎回指定する手間を省きます
