# 指示追従性

## tagline

LLM がプロンプトの細部まで忠実に従えるかを示す評価軸です。

<!-- ━━━━━━━━ 左ページ ━━━━━━━━ -->

## 何をしてくれるか

「です・ます調で、箇条書き 5 つ以内にまとめて」のような複数条件付き指示を、どれだけ取りこぼさず実行できるかを測ります。モデルの品質比較や用途選定の基準として使われます。

## どこで出会うか

LLM ベンチマーク記事やモデル比較表で「Instruction Following スコア」として目にします。プロンプトの条件が多いほど差が出やすく、実務でのモデル選びに直結する指標です。

## メイン図

### 図の狙い

同じ多条件プロンプトに対し、条件を守れたモデルと取りこぼしたモデルの出力を並べて差を示す。

### B. 登場シーン（figure_type: comparison）

- シーン1: 「5 つ以内・です・ます調・英語NG」の指示を各モデルに投げる
- シーン2: 条件を全部守ったモデルと 1〜2 条件を落としたモデルの出力を横並びで見せる
- シーン3: IFEval スコアの差がどこから来るかを矢印で指摘
- 並べる基準: 同一プロンプト・複数モデルの出力比較

## 会話での使い方例

「指示追従性が高いモデルほど、条件付きプロンプトの出力が安定しやすいです。」

## この用語の見どころ

### 1. 役割

複数条件の指示をどれだけ忠実に実行できるかの評価軸です。

### 2. うれしさ

モデル選びの際に「複雑な指示への適性」を数値で比較できます。

### 3. 注意点

スコアはプロンプト次第で変わるため、単一指標で判断しすぎない方が安全です。

### 4. どこで役立つか

ルール付き出力（フォーマット固定・文体指定）が多い業務での選定に役立ちます。

### 5. はじめに

IFEval などの評価ベンチマーク名とセットで押さえると理解が早いです。

### 6. 深掘り先

IFEval、MT-Bench、System Prompt

## 開発フローでの位置（必須）

1. 要件整理 — 出力形式・文体・件数などの条件を洗い出す
2. プロンプト設計 — 複数条件を System Prompt に明示して指定する
3. モデル比較 — 指示追従性スコアを参考に候補モデルを絞る
4. 出力検証 — 条件ごとの取りこぼしを目視またはスクリプトで確認する
5. 運用調整 — 取りこぼしが多い条件はプロンプト側で補強する

## 誌面ポンチ絵メモ

### メイン図（左ページ中段 / figure_type: comparison）

- 描く内容: 同じ多条件プロンプトを受け取った 2 人の人物（モデルを擬人化）が異なる出力を返す対比図
- 登場人物: モデル A（条件を全部守る）とモデル B（条件を 2 つ落とす）を表す人物 2 人
- 吹き出し・心の声: モデル A「5 条件、全部クリア！」モデル B「あれ、文字数は合わせたけど…」
- 中央に置くキーワード/ラベル: 「指示の条件チェックリスト」

## コミュニティ補完メモ

- G-4 System Prompt との住み分け：System Prompt はプロンプトの「書き方・置き場所」を扱うエントリ。本エントリは「モデルがその指示をどれだけ守れるか」という性能評価軸に絞る
- G-11 Context Engineering との住み分け：Context Engineering はコンテキスト全体の組み立て方を扱う。本エントリは出力の条件遵守という評価側の話
- E-4 HumanEval との住み分け：HumanEval はコード生成能力の評価ベンチマーク。本エントリは指示遵守能力の評価
- G-30 Tool Use との住み分け：Tool Use はツール呼び出し機能そのもの。本エントリは複数条件付き指示への対応能力という横断的評価軸
