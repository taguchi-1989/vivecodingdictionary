# Embedding

<!--

-->

## tagline

テキストや画像の意味を数列（ベクトル）に変換する技術。AI が「似ている」を数値で判断する土台です。

<!-- ━━━━━━━━ 左ページ ━━━━━━━━ -->

## 何をしてくれるか

文章や画像を数百〜数千次元のベクトル（数値配列）に変換します。似た意味ほど近い位置に集まるよう学習されており、意味の類似度を数値で比較できます。

## どこで出会うか

社内ドキュメントの意味検索や RAG 構築で登場します。OpenAI の text-embedding-3-small や sentence-transformers などの API で見かけます。

## メイン図

### 図の狙い

テキストがベクトルに変換され、類似ドキュメントが近くに集まる様子を視覚化します。

### C. 概念図（figure_type: structure）

- 中心に置く概念: Embedding モデル（変換器）
- 周辺の要素（3〜6個）: 入力テキスト群 / ベクトル空間（座標軸） / 類似ドキュメントのクラスター / クエリベクトル / 距離（類似度スコア）
- 関係の描き方（矢印・包含・比較）: テキスト → Embedding モデル → ベクトル空間へ矢印。類似したものが近くに丸まる様子をクラスターで示す。


## 会話での使い方例

「社内 FAQ を Embedding にしてベクトル DB に入れれば、自然な質問でも拾えます。」

## この用語の見どころ

### 1. 役割

テキスト・画像などのデータを意味付きの数値配列に変換します。

### 2. うれしさ

「意味の近さ」を数値で測れるため、曖昧な質問でも関連文書を拾えます。

### 3. 注意点

モデルごとに次元数・言語対応が異なるため、用途に合った選択が必要です。

### 4. どこで役立つか

RAG 構築・類似商品検索・クラスタリングなど、意味検索が要る場面全般です。

### 5. はじめに

「AI が言葉を数字の配列で記憶している」という理解で実務には十分です。

### 6. 深掘り先

RAG（G-15）、ベクトル DB（G-17）、Transformer（J-13）

## 開発フローでの位置（必須）

1. データ準備 — 社内ドキュメントや FAQ をテキスト形式に整える
2. Embedding 変換 — API（text-embedding-3-small 等）でベクトルに変換する
3. ベクトル DB 登録 — 変換済みベクトルをインデックスとして保存する
4. 類似検索 — クエリをベクトル化し、DB から近いものを取り出す
5. 結果活用 — 取り出した文書を LLM に渡して回答を生成する（RAG）

## コミュニティ補完メモ

- RAG（G-15）との住み分け：RAG は Embedding を使って検索した文書を LLM に渡す「仕組み全体」。Embedding はそこで使われる「変換技術」単体。
- ベクトル DB（G-17）との住み分け：Embedding は変換処理を指し、ベクトル DB はその変換結果を保存・検索するストレージを指す。セットで使われるが別の概念。
- Transformer（J-13）との住み分け：Embedding を生成するモデルの多くが Transformer ベース。深掘りしたい読者は J-13 へ。
