# 段階的開示

## tagline

情報を一度に全部出さず、必要な深さまで段階的に見せる設計原則です。

<!-- ━━━━━━━━ 左ページ ━━━━━━━━ -->

## 何をしてくれるか

「結論 → 根拠 → 例」「見出し → 要約 → 詳細」のように出力を階層化して、読み手が必要な深さまで自分で降りられるようにします。読みすぎ・読み飛ばしの両方を防ぐ情報整理の原則です。

## どこで出会うか

System Prompt（システムプロンプト）の設計やドキュメント生成の指示文を書くとき、「最初に箇条書きで概要を出して、詳細は後段に」と書き方を指定する場面で使います。アコーディオン UI との組み合わせで出力を折りたたむ実装でも意識されます。

## メイン図

### 図の狙い

「全部いきなり出す」状態と「階層で降りられる」状態を並べ、読み手の視線の動きを示します。

### B. 登場シーン（figure_type: structure）

- シーン1: ライターが LLM に「見出し → 要約 → 詳細」の順で書くよう指示する
- シーン2: 開発者が System Prompt に「最初に結論だけ 1 文、次に根拠 3 点」と書き込む
- シーン3: ユーザーがアコーディオン UI で必要な段落だけを開いて読む
- 並べる基準: 読み手が情報を受け取る粒度順

## 会話での使い方例

「段階的開示を意識した System Prompt にするだけで出力の読みやすさが変わります。」


<!-- ━━━━━━━━ 右ページ ━━━━━━━━ -->

## この用語の見どころ

### 1. 役割

LLM の出力を「読める構造」に整える設計原則です。

### 2. うれしさ

長い出力でも読み手が必要な箇所だけ拾えます。

### 3. 注意点

指示が曖昧だと階層が崩れて一塊の文章になりがちです。

### 4. どこで役立つか

ドキュメント生成・教育コンテンツ・レポート自動化で効果的です。

### 5. はじめに

「見出し → 要約 → 詳細」の順の指示から試すと感覚がつかめます。

### 6. 深掘り先

System Prompt、Context Engineering、プロンプトエンジニアリング

## 開発フローでの位置（必須）

1. 要件定義 — 「どの粒度まで自動生成するか」を決める段階で意識します
2. System Prompt 設計 — 「結論→根拠→例」のような出力構造を指定します
3. 出力レビュー — 階層が崩れていないかを確認して指示を調整します
4. UI 実装 — アコーディオンや折りたたみと組み合わせて体験を完成させます
