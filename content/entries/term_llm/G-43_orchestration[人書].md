# オーケストレーション

## tagline

複数のエージェント・ツールを司令塔が束ねて大タスクを完遂させる仕組みです。


<!-- ━━━━━━━━ 左ページ ━━━━━━━━ -->

## 何をしてくれるか

司令塔役のオーケストレーターが、専門エージェントやツールを必要に応じて呼び出し、結果を集約して次の指示を出します。1 体のモデルだけでは抱えきれない大規模タスクを、役割分担で完遂できます。


## どこで出会うか

Claude Code の Subagent 機能や、LangGraph・AutoGen・CrewAI といったフレームワークを使う場面で登場します。大規模リファクタや深い調査など、工程が多いタスクを自動化しようとしたときに名前を見かけます。


## メイン図

### 図の狙い

司令塔（オーケストレーター）が複数の専門エージェントを呼び分け、結果を束ねる構造を示す。

### C. 概念図（figure_type: structure）

- 中心に置く概念: オーケストレーター（司令塔エージェント）
- 周辺の要素: 調査エージェント、コード生成エージェント、レビューエージェント、Tool Use、結果集約
- 関係の描き方: 中心から矢印で各エージェントへ指示、返り矢印で結果を受け取る


## 会話での使い方例

「オーケストレーションで Subagent に分担させると大規模タスクが進みやすいです。」

## この用語の見どころ

### 1. 役割

複数エージェント・ツールを束ねて大タスクを完遂する司令塔の仕組みです。

### 2. うれしさ

1 体では無理な規模のタスクを役割分担で自動化できます。

### 3. 注意点

各エージェント間の依存関係が複雑になると障害の追跡が難しくなります。

### 4. どこで役立つか

大規模リファクタや深い調査など、多工程タスクの自動化に向きます。

### 5. はじめに

司令塔と専門エージェントの役割分担という概念を押さえると入りやすいです。

### 6. 深掘り先

Subagent、LangGraph、マルチエージェント協調


## 開発フローでの位置（必須）

1. タスク分解 — 大きなタスクをオーケストレーターが工程に分割します
2. エージェント呼び出し — 工程ごとに専門エージェントや Tool Use を起動します
3. 結果集約 — 各エージェントの出力をオーケストレーターが受け取ります
4. 判断・再指示 — 集約結果を見て次の行動を決め、必要なら再呼び出しします
5. 完了報告 — 全工程が終わったらユーザーに最終結果を返します

## コミュニティ補完メモ

- G-41 Subagent との住み分け：Subagent は「呼ばれる側の個体」を指す。オーケストレーションは「束ねる仕組み全体」を指すため、上位概念として扱う
- G-44 マルチエージェント協調との住み分け：マルチエージェント協調はエージェント同士が対等に連携する概念。オーケストレーションは司令塔が指示を出す階層型モデルを強調する
- G-30 Tool Use との住み分け：Tool Use は単一のツール呼び出し。オーケストレーションは複数の Tool Use やエージェントを組み合わせる上位の仕組み

## 備考

- LangGraph・AutoGen・CrewAI はフレームワーク名であり時変情報に当たるため、バージョン詳細は各フレームワークのエントリへ委ねる
- 「オーケストレーション」はソフトウェア全般（コンテナ orchestration 等）でも使われるが、本エントリでは AI エージェント文脈に限定する
