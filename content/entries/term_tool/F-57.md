# リポジトリ

## tagline

git（F-50）がソースコードを保管・履歴管理する単位です。プロジェクトフォルダに `.git` ディレクトリが加わった状態を指します。

<!-- ━━━━━━━━ 左ページ ━━━━━━━━ -->

## 何をしてくれるか

変更の履歴を丸ごと保持するフォルダ構造です。ワーキングツリー（実ファイル）、ステージング領域（コミット予定）、ブランチ・タグ・過去履歴をすべて 1 つの場所で管理します。


## どこで出会うか

Claude Code や Cursor を使い始めると「このリポジトリを見て」と AI に頼む場面が出てきます。GitHub（F-60）でプロジェクトを公開・共有するときも、リポジトリ単位でやり取りします。

### 図の狙い

「普通のフォルダ」と「リポジトリ」の違いを `.git` ディレクトリの有無で示す。


### C. 概念図（figure_type: structure）

- 中心に置く概念: リポジトリ
- 周辺の要素: .git（履歴データ）、ワーキングツリー（実ファイル）、ステージング領域、ブランチ、リモート（GitHub）
- 関係の描き方: リポジトリを中心に包含・矢印で示す

## この用語の見どころ

### 1. 役割

ソースコードと変更履歴を一元管理する入れ物です。

### 2. うれしさ

過去のどの時点にも戻れるので、試行錯誤が安心してできます。

### 3. 注意点

フォルダとの違いは `.git` の有無だけです。

### 4. どこで役立つか

AI への指示に「このリポジトリの構成に従って」と加えると精度が上がります。

### 5. はじめに

ローカルとリモートの 2 種類がある点を押さえます。

### 6. 深掘り先

git、GitHub、ブランチ


## 開発フローでの位置（必須）

1. 新規作成 — `git init` でフォルダをリポジトリに変換します
2. 複製 — `git clone` でリモートをローカルに取り込みます
3. 変更を記録 — ステージング → `git commit` で履歴を積みます
4. 共有 — `git push`（F-51）でリモートに送り、チームと同期します
5. 取り込み — `git pull`（F-52）でリモートの変更を手元に反映します

## コミュニティ補完メモ

- git（F-50）との住み分け：git はバージョン管理の仕組み・コマンド体系、リポジトリは git が管理するフォルダ単位。セットで理解する概念
- GitHub（F-60）との住み分け：GitHub はリモートリポジトリをホストするサービス。リポジトリの概念を先に理解してから GitHub を学ぶ順が自然
- .gitignore（F-56）との住み分け：リポジトリ内でトラッキング除外ファイルを指定するのが .gitignore。同じリポジトリ構造の話として関連する
