# JSON

## tagline

JavaScript Object Notation の略。波括弧と角括弧だけで構造化データを表現する形式です。

<!-- ━━━━━━━━ 左ページ ━━━━━━━━ -->

## 何をしてくれるか

テキストだけで「名前と値の組（オブジェクト）」や「値の並び（配列）」を表現し、異なるサービス・言語間でデータをやり取りできます。読み書きが簡単で、事実上の標準フォーマットです。

## どこで出会うか

Web API のレスポンスや、VS Code・Cursor などの設定ファイル（`settings.json`）でよく目にします。AI に「データを返して」と頼んだときも JSON 形式で返ってくることが多いです。

## メイン図

### 図の狙い

JSON の基本構造（オブジェクト・配列・プリミティブ）を入れ子で示し、「波括弧が名前と値の組、角括弧が並び」という関係を直感で掴んでもらいます。

### C. 概念図（figure_type: structure）

- 中心に置く概念: JSON オブジェクト `{ }` — 名前と値のペアが入る箱
- 周辺の要素: 文字列 `"text"`、数値 `42`、真偽値 `true`、配列 `[ ]`、入れ子オブジェクト `{ }`、`null`
- 関係の描き方: 外側の波括弧から矢印を内側の各型に向け、入れ子で階層を表す

## 会話での使い方例

「API のレスポンスが JSON なので、波括弧の中のキーを確認すれば値が取れます。」

## この用語の見どころ

### 1. 役割

言語に依存せずデータを構造化するテキスト形式です。

### 2. うれしさ

人間でも読みやすく、ほぼすべての言語で扱えます。

### 3. 注意点

コメントが書けず、末尾カンマも仕様上は無効です。

### 4. どこで役立つか

API 連携・設定ファイル・AI レスポンスの解析で使います。

### 5. はじめに

波括弧がオブジェクト、角括弧が配列の 2 点を押さえれば読めます。

### 6. 深掘り先

JSON Schema、JSONL、REST API。

## 開発フローでの位置（必須）

1. API 設計 — サーバーとクライアントのやり取りを JSON 形式で定義する
2. リクエスト送信 — ツールや AI に渡すデータを JSON で組み立てる
3. レスポンス受け取り — 返ってきた JSON の波括弧を読んでキーと値を確認する
4. 設定ファイル編集 — VS Code 等の `settings.json` を JSON で書き換える

## コミュニティ補完メモ

- YAML（F-7）との住み分け：YAML はインデントで階層を表す設定ファイル向け記法。JSON より人間が書きやすい反面、厳密な字句が異なる。データ交換には JSON、設定ファイルには YAML が選ばれることが多い。
- JavaScript（F-1）との住み分け：JSON は JavaScript の記法を元に設計されたが、現在は言語非依存の独立した規格（RFC 8259）。JavaScript 固有の機能（関数・コメント等）は含まない。
- REST API との住み分け：REST は API の設計スタイルで、レスポンスに JSON を使うことが多い。API 設計の詳細は REST のエントリへ。
- JSON Schema は JSON の構造を検証するための規格。本エントリでは触れず、深掘り先として案内する。
- JSONL（JSON Lines）は 1 行 1 JSON オブジェクトの形式で、ログ・AI 学習データでよく使われる。本エントリのスコープ外として深掘り先に留める。

## 備考

- JSON は Douglas Crockford が 2001 年頃に整理・普及させたデータ形式。RFC 4627 (2006) → RFC 7159 (2013) → RFC 8259 (2017) と更新されている。
- コメント非対応のため、設定ファイルで使う場合はコメントを書きたい要望が生じやすい。VS Code の `settings.json` は JSONC（コメント付き JSON）という独自拡張を採用している。
- 末尾カンマ（trailing comma）は標準 JSON では仕様違反だが、多くのパーサが許容している。厳密な仕様準拠が必要な場面では注意が必要。
