# git

## tagline

読み方は「ギット」です。コードの変更履歴を記録し、いつでも戻れるようにする仕組みです。

<!-- ━━━━━━━━ 左ページ ━━━━━━━━ -->

## 何をしてくれるか

「いつ・誰が・何を」変えたかを記録し、間違えても過去に戻せるバージョン管理ツールです。差分（diff）が行単位で見えるため、AI の提案を確認してレビューできます。

日々よく使うのは `add`（変更候補に入れる）・`commit`（記録）・`push`（送る）・`pull`（取ってくる）・`branch`（枝を分ける）の 5 つです。

## どこで出会うか

AI にコードを書かせる時代でも「何を受け入れ、何を戻すか」の意思決定基盤です。AI が壊しても `git restore` で戻せますし、commit しなければ AI の提案を受け入れない選択もできます。

差分が取れる → レビューできる → 人が責任を持てる、という連鎖は AI 駆動開発でも変わりません。Claude Code もブランチ → 実装 → commit → 差分レビューの往復を多用します。

## メイン図

### 図の狙い

「git なしで管理した世界」と「git で管理した世界」を左右に並べ、バックアップフォルダ散乱から 1 本の時系列へ、という構造変化を 1 枚で見せます。

### A. Before / After（figure_type: before_after）

- Before
  - 状況: git を使わず、バックアップを手動でコピーしながら編集
  - 視覚要素（コード or 概念）: `project_v1/` `project_v2/` `project_最新/` `project_本当の最新/` `project_本当の本当の最新/` が並ぶフォルダ群
  - つまずき: どれが最新か分からない。戻したい状態をどれか選べない。差分が見えないのでレビューできない
- After
  - 状況: git で全変更を 1 本の履歴として管理
  - 視覚要素: 縦一列に commit が並び、各 commit に「誰が、いつ、何を」のラベル。ブランチが枝分かれ／合流する
  - うれしさ: いつでも過去へ戻せる、差分が取れてレビューできる、並行開発も安心、AI に任せても戻せる

## 会話での使い方例

「AI に書かせても commit さえ刻んでおけば、いつでも `git restore` で戻せますよ。」

## この用語の見どころ

### 1. 役割

コード変更の時系列履歴を管理し、差分を取れるようにします。

### 2. うれしさ

差分が取れる。レビューできる。壊しても戻せる。共同作業と並行開発が成立します。

### 3. 注意点

コマンドが多く、最初はエラーメッセージに戸惑います。

### 4. どこで役立つか

共同開発、AI 駆動開発、長期保守のどれでも基盤になります。

### 5. はじめに

add／commit／push／pull／branch の 5 コマンドと差分の概念。

### 6. 深掘り先

rebase、cherry-pick、worktree、reflog、tag。

## 開発フローでの位置（必須）

1. コードを書く — 普通に編集する
2. `git add` → `git commit` — 変更を記録に残す
3. `git push` — リモート（GitHub）へ送る
4. レビュー → `git merge` — 差分を見て、人が責任を持って本線に取り込む

## 関連用語

- GitHub — git のリモートホスティング（git 本体とは別の会社サービス）
- branch — 並行作業のための枝
- commit — 変更を記録する単位
- merge — 別の枝を取り込む
- リポジトリ — git で管理されるプロジェクトの保管単位
