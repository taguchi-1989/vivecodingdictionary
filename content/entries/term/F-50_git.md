# git

## tagline

読み方は「ギット」です。コードの変更履歴を記録し、いつでも戻れるようにする仕組みです。

<!-- ━━━━━━━━ 左ページ ━━━━━━━━ -->

## ひとことで

git（ギット）は、コードの変更を時系列で記録し、複数人でも衝突せず共同作業できるようにするバージョン管理ツールです。

## 何をしてくれるか

「いつ・誰が・何を」変えたかを 1 行ずつ記録に残します。間違えても過去の状態に戻せます。複数人が同時に編集しても、あとで統合（merge）して一本の歴史にできます。

git の本質的なうれしさは **「差分（diff）が取れる」** ことにあります。どこがどう変わったか、行単位で見える。だから、レビューして受け入れるか戻すかを判断できる、AI の提案を確認できる、事故があっても追える、というすべてが成り立ちます。

日々よく使うのは 5 つです。

- `add` で変更を次のコミット候補に入れる
- `commit` で変更を 1 単位として記録
- `push` でリモート（GitHub など）へ送る
- `pull` でリモートの変更を取ってくる
- `branch` で並行作業の枝を分ける

## バイブコーディングでの位置づけ

AI にコードを書かせる時代でも、git は **「何を受け入れて、何を戻すか」の意思決定基盤** です。AI が壊しても `git restore` で戻せます。AI の提案を受け入れたくなければ commit しなければいい、と心理的な安全弁になります。

また、差分が取れる → レビューできる → **人が責任を持てる**、という連鎖があります。AI が書いたコードを現場に載せていく上で、この「責任の所在をはっきりさせる仕組み」は今後ますます必須になっていきます。エンジニアだけでなく、AI と一緒に物を作るすべての人にとって、git の基本概念は教養になっていくはずです。

Claude Code も実装中に git を多用します。ブランチを切る → 実装 → commit → 差分レビュー、という往復が AI 時代でも変わっていません。

## メイン図

### 図の狙い

「git なしで管理した世界」と「git で管理した世界」を左右に並べ、バックアップフォルダ散乱から 1 本の時系列へ、という構造変化を 1 枚で見せます。

### A. Before / After（figure_type: before_after）

- Before
  - 状況: git を使わず、バックアップを手動でコピーしながら編集
  - 視覚要素（コード or 概念）: `project_v1/` `project_v2/` `project_最新/` `project_本当の最新/` `project_本当の本当の最新/` が並ぶフォルダ群
  - つまずき: どれが最新か分からない。戻したい状態をどれか選べない。差分が見えないのでレビューできない
- After
  - 状況: git で全変更を 1 本の履歴として管理
  - 視覚要素: 縦一列に commit が並び、各 commit に「誰が、いつ、何を」のラベル。ブランチが枝分かれ／合流する
  - うれしさ: いつでも過去へ戻せる、差分が取れてレビューできる、並行開発も安心、AI に任せても戻せる

## 関連用語

- GitHub — git のリモートホスティング（git 本体とは別の会社サービス）
- branch — 並行作業のための枝
- commit — 変更を記録する単位
- merge — 別の枝を取り込む
- リポジトリ — git で管理されるプロジェクトの保管単位

## この用語の見どころ

### 1. 役割

コード変更の時系列履歴を管理し、差分を取れるようにします。

### 2. うれしさ

差分が取れる。レビューできる。壊しても戻せる。共同作業と並行開発が成立します。

### 3. 注意点

コマンドが多く、最初はエラーメッセージに戸惑います。

### 4. どこで効くか

共同開発、AI 駆動開発、長期保守、どれでも基盤になります。エンジニアに限らず必須の概念です。

### 5. 最初に理解する範囲

`add` / `commit` / `push` / `pull` / `branch` の 5 つと、「差分を取る」「責任の所在を残す」という役割。

### 6. 深掘り先

rebase、cherry-pick、worktree、reflog、tag。

## 開発フローでの位置（必須）

1. コードを書く — 普通に編集する
2. `git add` → `git commit` — 変更を記録に残す
3. `git push` — リモート（GitHub）へ送る
4. レビュー → `git merge` — 差分を見て、人が責任を持って本線に取り込む
