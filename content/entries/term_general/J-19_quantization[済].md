# 量子化

## tagline

モデルの重みを低ビット数で表現し、メモリと計算量を減らす技術です。


<!-- ━━━━━━━━ 左ページ ━━━━━━━━ -->

## 何をしてくれるか

LLM のパラメータは通常 FP16 で保存されます。量子化はこれを INT8 や INT4 に圧縮し、必要な VRAM を半分以下に減らせる場合があります。

## どこで出会うか

ollama や llama.cpp でモデル名に「Q4_K_M」「Q5_K_M」などの表記が並びます。Q4_K_M〜Q5_K_M がバランス推奨とされます。

## メイン図

### 図の狙い

精度ビット数と VRAM 消費量の関係を対比し、量子化を選ぶ判断軸を示す。

### B. 登場シーン（figure_type: comparison）

- シーン1: FP16 フルモデル — 高精度だが 70B なら 140 GB VRAM が必要
- シーン2: Q4_K_M 量子化 — 70B が 40 GB 前後に収まり RTX 4090 で動作可能
- シーン3: Q2 極端圧縮 — さらに小さいが精度低下が顕著でタスク依存
- 並べる基準: ビット数が下がるほど軽いがリスクも増す、というトレードオフ軸


## 会話での使い方例

「Q4_K_M 量子化で Llama 3 70B が手元で動いたので、API を呼ばずに済みました。」


<!-- ━━━━━━━━ 右ページ ━━━━━━━━ -->

## この用語の見どころ

### 1. 役割

モデル重みのビット数を下げてメモリを節約します。

### 2. うれしさ

一般 PC でも大型モデルをローカル動作できます。

### 3. 注意点

精度がわずかに落ちる場合があります（FP16 比 1〜5% 程度）。

### 4. どこで役立つか

ローカル LLM 構築や API コスト削減に役立ちます。

### 5. はじめに

Q4_K_M がバランス推奨という目安を覚えると入門に十分です。

### 6. 深掘り先

GGUF、AWQ、bitsandbytes

## 開発フローでの位置（必須）

1. モデル選定 — 動かしたい LLM とパラメータ数を決める
2. VRAM 確認 — 手元の GPU メモリに合う量子化レベルを選ぶ
3. モデル取得 — ollama や Hugging Face から量子化済みファイルを入手する
4. 推論実行 — ローカルで動作確認し、精度が許容範囲か評価する

## 備考

- Q4_K_M / Q5_K_M は GGUF 形式（llama.cpp 由来）の量子化レベル表記。「K」は k-quants（ブロック単位で量子化）を指し、「M」は Medium サイズの意味。同じ Q4 でも K_M と K_S（Small）では品質が異なる
- FP32（32 ビット）ではなく FP16 を基準としているのは、LLM の推論では通常 FP16 が実用精度の上限として扱われるため
- AWQ / GPTQ は GPU 向け量子化手法で、GGUF（CPU ＋ GPU 混在）とは適用環境が異なる。詳細は深掘り先へ
