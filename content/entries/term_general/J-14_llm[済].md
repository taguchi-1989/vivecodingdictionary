# LLM

## tagline

大規模言語モデルの略。文章で訓練された AI の主流で、主要 3 サービスの中身です。

<!-- ━━━━━━━━ 左ページ ━━━━━━━━ -->

## 何をしてくれるか

乱暴にまとめると「テキストを入れると、続きのテキストを返す」モデルです。骨格は「次に来る単語を予測する」だけですが、文章作成・要約・翻訳・対話・コード生成・画像説明と幅広い応用が生まれます。

## どこで出会うか

バイブコーディングの根幹で、「何を LLM に渡して、何を返してもらうか」の設計が出発点になります。モデルを変えるより Context を整えるほうが効く場面も多くあります。

## メイン図

### 図の狙い

LLM を中心に、入力・内部・出力の 3 層を 1 枚で見せます。「テキスト→予測→テキスト」という骨格と、Context／Token／パラメータ数／Transformer という周辺語彙の位置関係が分かるようにします。

### C. 概念図（figure_type: structure）

- 中心に置く概念: LLM（1 回の応答を生成する推論の箱）
- 周辺の要素（5〜6 個）: 入力側に「Context（文脈）」「Token（数え方）」／内部に「Transformer（アーキテクチャ）」「パラメータ数（B／T）」／出力側に「続きのテキスト」／下部に「訓練データ（インターネット規模）」
- 関係の描き方: 左から右へのフロー（Context → LLM → 出力）。下から LLM へ「訓練データ」の矢印。内部構造を点線の箱で示す

## 会話での使い方例

「LLM はテキストを入れて続きを返す箱で、Context の整え方で挙動がかなり変わります。」

## この用語の見どころ

### 1. 役割

文章を扱う AI の現代的な主流で、主要 3 サービスの中身です。

### 2. うれしさ

対話で使えて、幅広いタスク（文章・コード・要約・翻訳）に応用できます。

### 3. 注意点

Hallucination（事実と違う応答）の限界があり、万能ではありません。

### 4. どこで役立つか

文章作成、コード生成、要約、対話のほぼ全タスク。

### 5. はじめに

「次の単語を予測するモデル」の骨格と GPT／Claude／Gemini の系譜。

### 6. 深掘り先

Transformer、Attention、MoE、量子化、Fine-tuning、VLM。

## 開発フローでの位置（必須）

1. 目的を決める — 何を LLM にしてほしいか
2. 適切な LLM を選ぶ — 最も賢い／中間／アンカー、またはオープンモデル
3. Context を構成して渡す — 指示・資料・履歴
4. 出力を評価して反映 — 不足なら Context を足して再実行

## 非エンジニアのつまずき

- 「生成 AI」ではなく「LLM」と言うと、ちょっと気取って聞こえる気がして、口に出すのが少し気恥ずかしいです
- いまや画像も音声も扱えるマルチモーダルなのに、いまだに「言語モデル」と呼び続けるのに違和感があります
- 「次の単語を予測しているだけ」と説明されても、なぜそれで対話や要約までできるのかが直感に反していて、納得しきれません
