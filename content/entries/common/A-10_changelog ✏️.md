## 何をしてくれるか

AI 業界は 6 ヶ月でモデル名・料金プラン・サービス名が変わることがあります。本書は刷を重ねるごとに加筆・改訂する方針で、更新の対象と手順をここに記します。


## どこで出会うか

巻末または序文で「この情報はいつ時点か」が気になったとき、このページを参照してください。紙版を持っている場合は情報が古くなりがちなので、公式サイト（電子版）を併読すると執筆時点以降の改訂内容を確認できます。


## メイン図

### 図の狙い

「何が更新されるか」と「ID 体系は変わらない」という 2 点を対比で示す。

### C. 概念図（figure_type: timeline）

- 中心に置く概念: 改訂の対象範囲と ID 体系の不変性
- 周辺の要素: モデル名・料金・法令・著者体験・新興企業・ベンチマーク
- 関係の描き方: 「変わるもの」と「変えないもの」を左右に分けた対比図

## 会話での使い方例

「料金やモデル名の話は、執筆時点の情報なので公式サイトと併読が必要です。」

## この用語の見どころ

### 1. 役割

本書の改訂ポリシーと変更履歴をまとめた凡例です。

### 2. うれしさ

ID が変わらないので、旧版の参照番号も引き続き通用します。

### 3. 注意点

時変情報は evaluation_date を確認し、古い版では変わっている場合があります。

### 4. どこで役立つか

料金・モデル名など情報の鮮度が気になる場面で確認できます。

### 5. はじめに

「letter ID は変えない」「時変情報は evaluation_date つきで管理」の 2 点が基本です。

### 6. 深掘り先

A-6 評価日・時変情報の見方、A-3 図鑑の歩き方


## 開発フローでの位置（必須）

1. 刷行・版判断 — 改訂対象（モデル名・料金・法令など）が生じたか判断します
2. 該当エントリの evaluation_date を更新 — 時変情報の根拠日を引き直します
3. 本文末尾に「YYYY-MM 改訂」を追記 — 変更点を残します
4. changelog に要約を記録 — GitHub コミット履歴と巻末 changelog の 2 系統で残します
5. ID は動かさない — 追加は空き番号のみ、既存番号の意味は変えません
