# スキル：著者コメントを対話で回収する（fill-author-comment）

*著者の口述（音声入力混じりで雑な文）を、4 ラベル（first / good / bad / who）固定の「私のコメント」と 3 項目の「非エンジニアのつまずき」に整形して、エントリの所定位置に書き込むための手順書です。AI（Claude）が読む前提で書いています。*

## このスキルの目的

著者が PC 前で口述（または音声入力 → 文字起こし）した雑な発話を、誌面ルールに沿った形に整え、`<!-- user-input:start ... -->` ブロック内だけを書き換えることです。

**対話補助のみ**で、著者が触っていない既書きエントリの著者欄を AI 単独で埋めるのは禁止です。これは [skills/write-entry.md](write-entry.md) Step 5 「著者記入欄は空スケルトンのまま残す」の延長線上にあります。

## いつ使うか

- 著者が「F-50 の私のコメントを口述する」のように切り出した
- `ledgers/author_fill_queue.md` の未記入エントリを上から順に対話で潰すフェーズ
- 既存の著者欄を著者の意向で書き直す（リライト）

## 使わない場面

- AI 単独で著者欄を推測して埋める（→ 禁止）
- entry_writer / write-entry のスケルトン作成中（→ 空のまま残す）
- 本文（左右ページ）の修正（→ write-entry の Step 7 へ）

## 入力

- 対象 ID（例：`F-50`）
- 著者の口述テキスト（雑でよい、長文、文末がだ・である調になっていてもよい）
- 対象の節（`my_comment` ／ `stumble` ／ 両方）

## 出力

- `content/entries/{category}/{id}_{slug}*.md` の **`<!-- user-input:start key="..." -->` 〜 `<!-- user-input:end key="..." -->` の中だけ** を書き換え
- 書き換え対象キー：
  - `key="stumble"` ＝ 非エンジニアのつまずき
  - `key="my_comment"` ＝ 私のコメント
- そのほかの行（YAML、本文、裏台帳メモ）には触らない

## 確定ルール（毎回頭に置く）

[docs/v2_rules_summary.md](../docs/v2_rules_summary.md) §2-6 と [docs/entry_schema.yaml](../docs/entry_schema.yaml) の slot 定義から抽出（2026-05-02 改）。

### 私のコメント

| 項目 | 値 |
|:--|:--|
| 構造 | 4 ラベル固定 `🙂 第一印象` / `👍 良い点` / `👎 ダメな点` / `👥 誰向けか`（順序厳守）|
| ラベル絵文字 | `🙂` `👍` `👎` `👥`（誌面実装と一致）|
| 1 項目あたり字数 | 推奨 25〜40 字／許容 〜45 字／最低 10 字 |
| 合計字数 | 100〜180 字 |
| トーン | **です・ます調**（例外なし）|
| 上限超 | ⚠️ 警告（validator）|
| 書ける人 | 著者本人のみ。AI は対話補助 |

字数カウントは **ラベルとコロン（例：`🙂 第一印象: `）を除いた本文部分のみ**。

### 非エンジニアのつまずき

| 項目 | 値 |
|:--|:--|
| 構造 | 3 項目固定（箇条書き `- ...`）|
| 1 項目あたり字数 | 推奨 30〜50 字／許容 〜60 字／最低 15 字 |
| 合計字数 | 90〜180 字 |
| トーン | **です・ます調** |
| 上限超 | ⚠️ 警告 |
| 書ける人 | 著者本人のみ。AI は対話補助 |

「つまずき」は **「読者が引っかかりそうな点」ではなく「著者自身が実際につまずいた点」** だけを書きます。一次体験のないつまずきは作らない。

## 整形時の禁則

著者の口述を整える際、**意味を変えないこと**が最優先です。

- ❌ 主観を変える・断定を強める・弱める
- ❌ 著者が言っていない事実を補う（例：「2024 年に〜」を勝手に足す）
- ❌ だ・である調のまま書き込む（→ 必ずです・ます調へ変換）
- ❌ ラベルの順序を入れ替える
- ❌ ラベルの絵文字を勝手に変える（仕様書 §195 と現行誌面に乖離があるが、実装は `🙂👍👎👥`）
- ❌ 4 ラベル以外のラベルを追加する
- ❌ 強い断定語（最新／最強／完全／絶対／必ず）を入れる
- ⭕ 助詞の調整、語順の整え、重複の圧縮、口語の冗長表現を削る、です・ます語尾化

迷ったら、著者の言葉そのままに近づける方を選ぶこと。

## 対話の流れ

### Step 0. 対象を特定する

著者の切り出しから ID（例：`F-50`）と対象節を判定。曖昧なら確認する。

### Step 1. 当該エントリを読み込む

```
Read content/entries/{category}/{id}_{slug}*.md
```

`user-input:start key="..."` ブロックの中身（既存記入の有無）を確認。

- 既存記入があれば、著者が「上書き」か「追記」かを確認する
- 既存記入の文体・ですます感・字数も参考にする

### Step 2. 口述を 4 ラベル（or 3 項目）に振り分ける

著者の発話を読みながら、どの文がどのラベルに該当するか頭の中で振り分ける。

**私のコメントの 4 ラベル早見**：

- 🙂 第一印象 ＝ 最初に触れたとき／名前を聞いたときの感触、戸惑い、馴染みのなさ
- 👍 良い点 ＝ 使ってよかった点、AI 駆動開発での効きどころ、別格な体験
- 👎 ダメな点 ＝ 使いづらさ、説明ハードル、非エンジニアにとっての敷居
- 👥 誰向けか ＝ おすすめしたい層、本書のどの読者層に響くか

**つまずきの 3 項目早見**：

- 読み方／呼び方／名前のつまずき
- 過去の失敗エピソード（戻せなかった／壊した／消した）
- 用語・コマンドの多さ・概念の重なり

口述に該当ラベルの素材が足りない場合：**勝手に補わない**。著者にその場で聞く。

### Step 3. 整形案を出す（書き込み前）

整形した 4 行（または 3 行）を **マークダウンの箇条書きで提示** し、各行末に `（NN 字）` で字数を併記する。

```
- 🙂 第一印象: 読み方が「ギット」か「ジット」か分からず...しませんでした。（43 字）
- 👍 良い点: これがないと LLM に...別格です。（38 字）
- 👎 ダメな点: エンジニア以外への...押し寄せます。（36 字）
- 👥 誰向けか: AI を使いこなしたい...全然違います。（36 字）

合計 153 字（範囲 100〜180）／各項目 45 字以内／です・ます調
```

著者の確認を取ってから書き込む。**確認なしで書き込まない**。

### Step 4. 字数チェックを必ず通す

書き込み前に以下を機械的にチェック：

1. 1 項目あたり ≤ 45 字（コメント）／ ≤ 60 字（つまずき）
2. 合計が 100〜180 字（コメント）／ 90〜180 字（つまずき）
3. 全行が `です・ます` 語尾で終わる
4. ラベル順 🙂 → 👍 → 👎 → 👥（コメント）
5. ラベル絵文字が `🙂👍👎👥` で、`★✓!☺` ではない
6. 強い断定語が入っていない

1 つでも外れたら案を作り直し、Step 3 から再提示。

### Step 5. 書き込む

`Edit` ツールで `<!-- user-input:start key="..." -->` と `<!-- user-input:end key="..." -->` の **間だけ** を置換する。マーカー行自体は触らない。

私のコメントの例：

```markdown
<!-- user-input:start key="my_comment" -->
- 🙂 第一印象: ...
- 👍 良い点: ...
- 👎 ダメな点: ...
- 👥 誰向けか: ...
<!-- user-input:end key="my_comment" -->
```

つまずきの例：

```markdown
<!-- user-input:start key="stumble" -->
- ...
- ...
- ...
<!-- user-input:end key="stumble" -->
```

### Step 6. 保存後の挙動を確認

- `Edit` 完了で PostToolUse hook の `validate_entry.py` が走り、`update_review_queue.py` が `ledgers/revision_queue.md` を再生成
- 4 項目（コメント）／3 項目（つまずき）が両方埋まると `drafting → needs_review` への自動昇格条件を満たす
- ただし `needs_review → ready` は著者本人の手動操作（自動昇格しない）

### Step 7. 次のエントリへ

`ledgers/author_fill_queue.md` を再読込し、未記入のうち次の ID を提示。著者が「上から順」「グループ別」「飛び石」のいずれかを指定する。

## 字数カウントの厳密ルール

- 半角英数 1 字 ＝ 1 字
- 全角 1 字 ＝ 1 字
- 絵文字 1 字 ＝ 1 字（ただしラベル絵文字 `🙂` `👍` `👎` `👥` は本文字数に**含めない**）
- ラベル `🙂 第一印象: ` などの部分は字数に**含めない**
- 句読点・スペース・括弧は含める

Python での目安計算：

```python
text = "読み方が「ギット」か「ジット」か分からず、英語ではなく造語なので必要性が腹落ちしませんでした。"
print(len(text))  # → 43
```

## よくある失敗と対処

| パターン | 対処 |
|:--|:--|
| 口述がだ・である調 | 全文をです・ます調に書き換えてから整形 |
| 第一印象が長くなる（60 字超） | 「なぜ必要か」「最初に出会った場所」のどちらか 1 つに絞る |
| 良い点が抽象的（「便利」のみ） | AI 駆動開発・LLM 文脈での具体に寄せられないか著者に問う |
| 誰向けかが「全員」止まり | 「特に〜したい人」の補強を著者に問う |
| つまずきが 4 項目以上ある | 3 つに絞ってもらう。残りは「備考」へ |
| つまずきが 1〜2 項目しかない | 3 つ目を著者に問う。作らない |

## 守ること（テイストの核）

- **対話補助に徹する**：AI 単独で埋めない
- **意味を変えない**：助詞と語尾調整に留める
- **です・ます調**：例外なし
- **字数上限を守る**：合計 100〜180／180 を超える整形案は提示しない
- **書き込み前に必ず確認**：著者の OK なしに `Edit` しない
- **マーカーの外を触らない**：`user-input:start/end` の間だけ書き換え

## やらないこと

- 本文（左右ページ）の修正 → [skills/write-entry.md](write-entry.md)
- 著者欄の **AI 単独補完** → 禁止
- ステータス変更（`needs_review → ready`）→ 著者本人のみ
- 著者欄の絵文字変更や項目追加・削除 → 仕様変更案件、別議論

```
fill-author-comment F-50
- 対象節: my_comment
- 著者口述（生）: 「まず名前がじっとのかぎっとのかっていうのでこれ読み間違いなかったな...」

→ Step 1 で F-50_git[済].md を読み、Step 2 で 4 ラベルに振り分け、Step 3 で字数付き案を提示、確認後 Step 5 で書き込み
```

```

```
