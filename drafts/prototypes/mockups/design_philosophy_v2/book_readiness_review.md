# 書籍化レベル 品質レビュー（iter 15 時点）

*2026-04-24。TypeScript 見開きプロトタイプ（`typescript_spread.html`）を「このまま紙で刷って流通に乗せられるか」の観点で点検したもの。HTML/CSS 技術的品質ではなく、**本として店頭で手に取ったときの違和感**の棚卸し。*

---

## 1. 寸法・印刷工程の観点

### 縦横比
- 現状: **750 × 1080px、比率 1:1.44**
- ISO A/B 系 (A4/A5/B5/B6 など全て): **1:1.4142 (√2)** ← 日本の図鑑・書籍の大多数
- 現状は **約 2% 縦長**。見開き 2 ページでは合計 4% の縦伸びとなり、紙面を刷ると「細長い」違和感が出る
- **修正案**: `min-height: 1080 → 1061` で √2 ぴったり

### 塗り足し（bleed）
- 未定義。`.top-bar` の 4px 青バーが紙端まで到達する場合は、印刷用 PDF に 3〜5mm の bleed を足す必要あり
- 現プロトタイプは画面表示用。印刷版は別ファイルで bleed 込みに書き直す前提

### ノド（gutter）余白
- 現状、左右ページとも `padding: 40px 44px 100px`（対称）
- **書籍では非対称が常道**: 左ページは右padding を多め（ノド側）、右ページは左padding を多めにする
- 見開きで中央付近の文字がノドに食われて読みにくくなる可能性
- **修正案**: 左ページ `padding: 40px 56px 100px 44px`、右ページ `padding: 40px 44px 100px 56px` 程度

### 解像度
- Web mockup は 750px 幅で作成済。A4 相当を 300dpi で印刷するなら 2480px 幅が必要
- ベクターの SVG は問題なし、フォントは TTF 埋め込みで解像度フリー
- **画像素材（写真・イラスト）を今後入れる場合は 300dpi 以上で用意する**

### 色空間
- CSS は RGB で `#2F7BE6`（ink-blue）を定義
- 印刷は CMYK。特にこの青は**彩度が落ちやすい色域**
- 色校正 PDF 出力時、CMYK 変換後の青が沈むリスク。**本刷り前に色校を必須**

## 2. 読み心地・視覚品質の観点

### 情報密度の左右差（△）
- 右ページ: 6 視点 + つまずき + コメント + フロー = **高密度**
- 左ページ: タグライン + 2 セクション + Before/After + 関連用語 = **関連用語の下に空白**
- 現状はガラガラな左下がもったいない。**将来の画像スロット**（ユーザー言及）で埋める前提
- 解消前はアンバランス感が残る

### 図の「コード直出し」問題（△）
- Before/After の中身は `function add(a, b) { return a + b }` のソースコード
- **非エンジニア読者の目には威圧的**
- 全 335 entry のうちプログラミング系はこのパターンが多い。代替アプローチ:
  - **概念図**（図鑑では `figure_type: structure / comparison`）を多めに使う
  - **イラスト**（ポンチ絵メモに計画を書く運用）で「コードのズレを人物が気づく」シーンにする
  - コード直出しは「ここは技術者向けの節です」と読者に明示するアイコン記号（例: `</>`）で導く
- Design System 側でこの配慮が明文化されていない

### 実写・手描きイラストの不在（△）
- プロトタイプは全部、CSS 描画の幾何学的要素（アイコン、罫、枠、塗り）
- 良い本は **章扉や要所にイラスト or 写真** がある。「哲学用語図鑑」路線なら、各 entry に「使う人」のイラストが 1 枚ほしい
- CLAUDE.md の図解方針（人物＋吹き出しで「使う人がどう感じるか」）と合致するが、**実際に絵を入れる場所が本レイアウトにまだ無い**

### 単調さ（335 entry 並ぶと）（△）
- 全 entry がまったく同じレイアウト・同じ色・同じ節構成
- 読者は 30 entry 見た頃に「見たことある感」で疲れる
- 対策案:
  - **章ごとのアクセントカラー** — でも Design System は青単色縛り。章ごとの図柄パターン変化で代替？
  - **章扉に 1 ページ絵** を挟む
  - **節見出しの図柄を entry によって変える**（6 視点アイコンの配置や色をカテゴリで変える）
  - **「関連用語マップ」** などの変化球ページを章末に配置

### ナビゲーション補助の不在（△）
- 巻頭の目次 + 巻末の索引があるとして、**本文ページには章インデックスが無い**
- 厚い書籍では「天」「地」「小口」に章区切りの色帯や文字を配置する
- 現プロトタイプの top chrome は「技術用語」「TypeScript の見方」だが、**どの章の何エントリ目か** が手に取って分かりにくい
- 修正案: 右ページ右端（フォアエッジ）に章記号の細いバー、ページ番号の前に章番号

## 3. タイポグラフィの細部（△-○）

### 行長
- 左ページ本文 「何をしてくれるか」「どこで出会うか」: 約 36〜40 文字／行 → 和文の読みやすい範囲（40-50）の下限。**OK**
- 右ページ 6 視点 body: 14〜17 文字／行 → 短すぎて改行が多い。**セル幅の制約で仕方なし**
- Before/After コード枠: 約 27 文字 → コードなので OK

### ぶら下げ（burasage / hanging punctuation）
- 未設定。CSS `hanging-punctuation` は Safari / Firefox でサポートあり Chrome は部分サポート
- 和文組版の品格のためには設定しておきたい。現在「、。」が行頭に落ちる可能性あり

### palt（プロポーショナル詰め）
- base.css で `font-feature-settings: "palt"` 適用済 ✓
- Zen Maru Gothic は丸みの連続が詰めすぎると読みにくい。palt を弱める選択肢もあり（半角英数との混植時のみ有効化など）

### フォント埋め込み
- HTML は `fonts/ZenMaruGothic-*.ttf` をサーバから配信
- PDF 化する際は **サブセット埋め込み必須**（ライセンス: SIL OFL、埋め込み可能）
- 印刷会社のワークフローと相性確認

## 4. コントラスト／可読性（○）

- 本文 `--ink` (#1A1A1A) on `--paper` (#FFFFFF): **16.9:1** WCAG AAA 楽勝
- タグライン `--ink` on `--ink-blue-100` (#D6E4F8): **12.1:1** AAA
- 見出し `--ink-blue-900` (#123E82) on `#FFFFFF`: **10.9:1** AAA
- 6 視点 body `--ink-2` (#4A5568) on `#FFFFFF`: **8.1:1** AAA
- 薄い補足 `--ink-3` (#8A94A6) on `#FFFFFF`: **3.8:1** ← **AAA 下回る**（4.5:1 が閾値）。補足情報用途だが、必須情報には使わない方針を明記すべき

## 5. 総評（5 段階）

| 観点 | 評価 | 備考 |
|---|---|---|
| 誌面の秩序・美しさ | **◎** | 青単色＋白紙＋罫 の規律が効いている |
| 情報の階層明示 | **◎** | 左青バー、番号バッジ、アイコン、すべて役割を果たしている |
| 非エンジニア読者への親切さ | **○** | 「です・ます」維持、但しコード直出しはまだ威圧的 |
| 印刷工程への適応 | **△** | 縦横比 2% ずれ、bleed 未定義、gutter 未対応、CMYK 校正未実施 |
| 量産時の単調さ回避 | **△** | 全エントリ同型のため、中盤以降の飽きが課題 |
| 書籍としての完成度 | **○** | プロトタイプとしては十分高いが、刷る前にやることが上記 4 項目あり |

## 6. 次の優先タスク（書籍化に向けて）

1. **縦横比を 750×1061 に修正**（今すぐできる、影響小）
2. **ノド非対称 margin を導入**（今すぐできる、影響中）
3. **実イラスト／写真の入れ場所を設計**（左ページ余白、章扉、entry 脇）
4. **コード直出しの代替表現ガイドライン**（non-engineer 向けの entry では「図」にするルール）
5. **章インデックスの誌面表示**（フォアエッジ帯、章番号）
6. **CMYK 色校正**（実刷り前必須）
7. **章ごとの視覚的変化ポイントを仕込む**（同型疲労の緩和）

### 優先度低（運用で吸収できる）
- ぶら下げ設定の明文化
- `--ink-3` を必須情報に使わない運用ルール化
- 300dpi 画像素材の用意（画像入れ始めたら）
