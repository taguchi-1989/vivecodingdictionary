# 誌面ワイヤーフレーム案

## 共通前提
- 1項目1ページを基本にする。
- 白背景、青系2色、グレー1色を基本にする。
- 図は主役だが、図の中の文字は少なくする。
- 本文は「辞書引き」できる短いブロックに分ける。
- 体験区分、読者レベル、評価日は小さくても必ず見える位置に置く。

## 共通パーツ

### ヘッダー
- ID
- category / subtype
- title
- subtitle または ひとことで

### メタ情報
- experience_level
- reader_level
- evaluation_date
- status

### 本文ブロック
- 何をしてくれるか
- 実務での意味
- 非エンジニア視点のつまずき
- 最初にどこまで理解すればよいか
- 私のコメント

### フッター
- related_terms
- source note
- update risk label

## Wireframe A: 図鑑カード型

向いている項目:
- Service
- Tool
- Person
- 入口となる大きな概念

対象例:
- Gemini
- Cursor
- Vercel

```text
┌──────────────────────────────────────────────┐
│ ID 101 / service / ai_assistant      Lv.2    │
│ Gemini                                       │
│ Googleの生成AIサービスとモデル群の名前       │
├──────────────────────┬───────────────────────┤
│                      │  何をしてくれるか      │
│      STRUCTURE       │  実務での意味          │
│       FIGURE         │  つまずき              │
│                      │                       │
├──────────────────────┴───────────────────────┤
│ 私のコメント                                  │
│ 関連語: Gemini 2.5 Pro / Vertex AI / API      │
│ checked: 2026-04-21                           │
└──────────────────────────────────────────────┘
```

判断ポイント:
- 一覧性が高い。
- 図鑑らしい。
- 長文には弱いので、説明は短く切る。

## Wireframe B: 比較マップ型

向いている項目:
- Model
- Benchmark
- 複数候補の比較

対象例:
- Gemini 2.5 Pro / Flash / Flash-Lite
- Claude Sonnet 系比較
- SWE-Bench と Terminal-Bench

```text
┌──────────────────────────────────────────────┐
│ ID 201 / model                       Lv.2    │
│ Gemini 2.5 Pro / Flash / Flash-Lite          │
│ 用途別に分かれたGeminiのモデル群             │
├──────────────────────────────────────────────┤
│              2 AXIS COMPARISON MAP           │
├──────────────┬──────────────┬────────────────┤
│ Pro          │ Flash        │ Flash-Lite     │
│ 深く考える   │ バランス型    │ 軽く大量に回す │
├──────────────┴──────────────┴────────────────┤
│ 選び方 / つまずき / 私のコメント              │
│ checked: 2026-04-21 / update risk: high       │
└──────────────────────────────────────────────┘
```

判断ポイント:
- 似た名前の違いを説明しやすい。
- 数値を入れると更新負荷が上がる。
- 初版では「判断軸」を主役にするほうがよい。

## Wireframe C: Before / After 型

向いている項目:
- Term
- つまずきが明確な技術用語
- 導入前後で意味が見える概念

対象例:
- TypeScript
- ESLint
- Formatter

```text
┌──────────────────────────────────────────────┐
│ ID 302 / term / language              Lv.2   │
│ TypeScript                                    │
│ JavaScriptに型の考え方を足す言語              │
├──────────────────────┬───────────────────────┤
│ BEFORE               │ AFTER                 │
│ 実行するまで気づきにくい│ 実行前に警告が出る   │
├──────────────────────┴───────────────────────┤
│ 実務での意味                                  │
│ 非エンジニア視点のつまずき                    │
│ 私のコメント                                  │
└──────────────────────────────────────────────┘
```

判断ポイント:
- 「何が嬉しいか」が一番伝わりやすい。
- 正確さを落としやすいので、本文側で補足する。
- 技術用語ページの標準候補にしやすい。

## Wireframe D: ワークフロー型

向いている項目:
- Workflow
- Framework
- Tool chain
- 処理の流れで理解する項目

対象例:
- Next.js
- GitHub Actions
- CI/CD

```text
┌──────────────────────────────────────────────┐
│ ID 305 / term / framework             Lv.2   │
│ Next.js                                       │
│ ReactでWebアプリを作るためのフレームワーク    │
├──────────────────────────────────────────────┤
│ Components → Routing → Server → Build → Deploy│
├──────────────────────────────────────────────┤
│ 何をしてくれるか                              │
│ 実務での意味                                  │
│ つまずき                                      │
│ 関連語: React / TypeScript / Vercel           │
└──────────────────────────────────────────────┘
```

判断ポイント:
- 開発工程やツールチェーンに強い。
- 図が横長になりやすい。
- 印刷では折り返しレイアウトも必要。

## 初期採用案
まずは3つを標準にする。

1. `structure`: 図鑑カード型
2. `comparison`: 比較マップ型
3. `before_after`: Before / After 型

`workflow` と `timeline` は、項目が増えてから正式化する。
