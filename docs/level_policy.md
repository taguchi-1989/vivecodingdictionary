# レベル運用ポリシー v1（2026-05-27）

3 つの「レベル」軸 — `experience_level` / `reader_level` / `importance` — の **判定基準** を 1 枚にまとめたものです。

[docs/entry_schema.yaml](entry_schema.yaml) と [docs/v2_rules_summary.md](v2_rules_summary.md) は「どの値が許容されるか（enum）」を定義します。本ドキュメントは「**どの値を選ぶか**」を定義します。執筆・レビュー時はこのファイルだけを開けば判断できるようにしました。

---

## 0. 3 軸の関係

| 軸 | 何を測るか | 主体 | 用途 |
|---|---|---|---|
| `experience_level` | 著者の触り具合（事実） | 著者 | 本文の断定度・主観の信頼度のバッジ |
| `reader_level` | 読み解くのに必要な前提知識の深さ | 読者（横軸） | 「これを読むには事前に何を知っている必要があるか」 |
| `importance` | 本書全体での読み始め優先度 | 読者（縦軸） | 「優先して読むべきか」 |

**相関はあるが独立軸**として扱います。たとえば「ノイマン型コンピュータ」は `importance: E`（研究者向け）でも `reader_level: 2`（用語そのものは素朴に説明可能）になり得ます。逆に「Hook」は `reader_level: 3`（運用知識が前提）だが `importance: C`（業務本格活用層が読む）です。

---

## 1. `experience_level`（著者軸）

著者（応用情報・G 検定保持、機械系開発業務、AI を使う側）が **その対象を自分の手・目・耳でどれくらい触れたか** を示すバッジです。本文の断定度の信頼度を読者に開示する目的で付けます。

### 1-1. 判定フロー

```
対象は何か？
├─ プロダクト／ツール／モデル
│   ├─ 30 分以上、自分のアカウントで動かしたことがある              → hands_on
│   ├─ 同シリーズ／同社の別プロダクトは触ったが当該はサイトで眺めただけ → partial
│   └─ 公開資料・ニュース・他人のデモ動画のみ                         → research_only
│
├─ 人物
│   ├─ 著者本人 ／ 直接会話・取材したことがある                       → hands_on
│   ├─ その人の本・講演・配信を 30 分以上見た／読んだ                 → partial
│   └─ ニュース・第三者記事のみ                                       → research_only
│
├─ 組織・企業
│   ├─ その組織のプロダクトを業務で使っている                         → hands_on
│   ├─ 関連プロダクトを触った／公式ドキュメントを読み込んだ           → partial
│   └─ ニュース・第三者記事のみ                                       → research_only
│
├─ 出来事・歴史
│   ├─ リアルタイムで体験した（同時代に当事者として接触）             → hands_on
│   ├─ 直後の報道・関連プロダクトを触って間接体験                     → partial
│   └─ 後年に資料で知っただけ                                         → research_only
│
└─ 概念・用語
    ├─ 業務で日常的に使っている／触る前提で本書を書いている           → hands_on
    ├─ 言葉として知っているが業務で使ったことはない                   → partial
    └─ 本書執筆のために調べた                                         → research_only
```

### 1-2. 注意

- **本文の断定度と一致させる**。`research_only` なのに「最強」「事実上の標準」と書かない。
- **時間で変わる**。半年後に触ったら `partial → hands_on` に上げ直す（`evaluation_date` も更新）。
- 判定に迷ったら一段下げる（誇張より控えめが安全）。

## 2. `reader_level`（前提知識軸）1〜6

その用語を **誌面の本文だけで理解する** ために、読者がどの程度の前提知識を持っている必要があるかを示します（**ハードル基準**）。**用語の難しさ** ではなく **読者の負担** です。

**6 段階を「3 グループ × 2 段」で構造化** しています。グループの境目はバイブコーディングの読者像に沿って質問 1 つで切ります。

### 2-1. 3 グループ × 2 段の定義

| グループ | 値 | 想定読者 | この値で書けるとは | 例 |
|---|---|---|---|---|
| **Ⅰ. 一般** | **1** | AI に触れたことがない／名前を聞いた程度 | 専門用語ゼロ | まえがき, AI（一般語） |
|  | **2** | AI チャットを月に数回は使う | 「ChatGPT」「LLM」が既知 | ChatGPT, LLM, Gemini, Hallucination |
| **Ⅱ. 業務で AI を使う非エンジニア** | **3** | 業務で AI を週 1 回以上使う | プロンプト・API・モデル が説明なしで通じる | MCP, Tool Use, Token, Context, Deep Research |
|  | **4** | JSON/YAML/CLI を見て自分で書き換えられる | 設定ファイル・コマンドラインが通じる | .claude/settings.json, Hook, Slash Command, Subagent |
| **Ⅲ. 実装・運用・関与する人** | **5** | AI を組み込んだ業務・ツールを設計し回す | **自分でゴリゴリ書かなくていい**。コードを読み、AI に書かせて動かす | LoRA, Tauri, LangChain, Whisper, MoE |
|  | **6** | ガバナンス・規制・モデル開発・安全性まで踏み込む | 論文・規制動向・組織運営まで読む | gpt-oss, ノイマン型, Constitutional AI, EU AI Act, Anthropic 創業 |

### 2-2. 判定フロー（質問 3 つで一意に切る）

```
Q1. 業務で AI を週 1 回以上使う前提か？
  ├─ NO  → グループⅠ
  │   └─ Q1a. AI チャットを 1 回でも触ったことが前提か？
  │       ├─ YES → 2
  │       └─ NO  → 1
  │
  └─ YES → Q2. 自分で書かなくていい、コードを読んで AI に書かせる前提か？
        ├─ NO  → グループⅡ
        │   └─ Q2a. 設定ファイル（JSON/YAML）を開いて編集する手順が出るか？
        │       ├─ YES → 4
        │       └─ NO  → 3
        │
        └─ YES → グループⅢ
              └─ Q3a. AI のガバナンス・規制・モデル開発・安全性まで踏み込むか？
                  ├─ YES → 6
                  └─ NO  → 5
```

### 2-3. グループの境目の決め手（要点）

| 境目 | 決め手の質問 |
|---|---|
| **Ⅰ ⇔ Ⅱ**（2 ↔ 3） | 業務で AI を週 1 回以上使うことが前提か？ |
| **Ⅱ ⇔ Ⅲ**（4 ↔ 5） | **自分で書かなくていい**。コードを読み AI に書かせて動かす前提か？ |
| Ⅰ内（1 ↔ 2） | AI チャットを 1 回でも触った前提か？ |
| Ⅱ内（3 ↔ 4） | 設定ファイル（JSON/YAML）の編集手順が出るか？ |
| Ⅲ内（5 ↔ 6） | ガバナンス・規制・安全性まで踏み込むか？ |

### 2-4. 範囲表記（`"2-3"` のような表記）

| 書き方 | 意味 | 推奨 |
|---|---|---|
| `2` | 単一値（最も明快） | **第一選択** |
| `"2-3"` | グループ Ⅰ→Ⅱ をまたぐ。入口は 2、深掘りは 3 | グループ境目をまたぐときに限り使う |
| `"4-5"` | グループ Ⅱ→Ⅲ をまたぐ。入口は 4、深掘りは 5 | 同上 |
| `"5-6"` | グループⅢ内、Ⅲ-1 段 vs Ⅲ-2 段の幅 | 内部の幅で迷うときのみ |
| `"1-2"` / `"3-4"` | グループ内の幅 | **可能なら単一値に寄せる**（迷い表記） |
| `"2-4"` 以上 | **使用禁止**（2 段以上の幅） | 単一値に寄せる |

**表記ルール**：
- 単一値はクォートなし（`reader_level: 2`）
- 範囲値はクォート必須（`reader_level: "2-3"`）— `2-3` だけだと YAML が日付と誤解する
- 範囲幅は最大 1 段（`"2-3"` / `"4-5"` / `"5-6"`）

### 2-5. 「ハードル基準」を固定する

reader_level は **「読み手にこれだけの前提知識が必要」（ハードル基準）** で固定します。「この用語まで読み進める人はふつうこのレベル」（到達層基準）ではありません。

例：H-53 ChatGPT 登場 → ハードル基準では **1**（誰でも読める）。到達層基準なら 2〜3（歴史を読む人）ですが、本書では前者で運用します。

## 3. `importance`（優先度軸）A〜E

**誰がまず読むべきか** の優先度。本書は「AI を仕事で使いたい非エンジニア」が読者像なので、その人にとっての必要度を基準にします。

### 3-1. 5 段階の定義

| 値 | 想定読者 | 「読まないと困る」場面 | 例（schema 既存定義から） |
|---|---|---|---|
| **A**（必須） | AI ニュースを目にする全員 | テレビ・新聞・SNS で出てくる | ChatGPT, Claude, Gemini, LLM, Hallucination |
| **B**（一般） | 仕事で AI を使う非エンジニア | 業務で AI ツールを選ぶ・触る | Token, Context, MCP, Cursor, Deep Research |
| **C**（中級） | 業務で本格活用する人 | AI を組み込んだ業務を回す | Tool Use, Hook, Slash Command, Subagent |
| **D**（上級） | エンジニア・実装者 | 自分で実装・チューニング | 量子化, LoRA, Tensor コア, Tauri, Whisper |
| **E**（開発者向け） | 研究者・歴史・コア実装 | 背景・系譜・原典を追う | gpt-oss, ノイマン型, Anthropic 創業 |

### 3-2. 判定フロー（5 段階を一意に決める）

```
Q1. その用語を知らないと、AI ニュースの見出しが読めないか？
  ├─ Yes → A
  └─ No  → Q2

Q2. その用語を知らないと、業務で AI を使うとき選択肢を間違えるか？
  ├─ Yes → B
  └─ No  → Q3

Q3. その用語を知らないと、AI を業務に組み込む設計ができないか？
  ├─ Yes → C
  └─ No  → Q4

Q4. その用語を知らないと、AI を自分で実装・拡張できないか？
  ├─ Yes → D
  └─ No  → E（背景・歴史・コア実装）
```

### 3-3. ありがちな誤判定

- ❌ **「自分が知っているから A」** → 読者像は自分ではない。基準は「非エンジニアが業務で困るか」
- ❌ **「むずかしい用語だから E」** → 難しさは reader_level 側。importance は「読む必要」
- ❌ **「人物だから一律 C」** → 経営者 Sam Altman/Jensen Huang はニュースで頻出（B）。研究者 Yann LeCun は背景（D）
- ❌ **「歴史だから一律 E」** → ChatGPT 登場（H-53）はニュースで誰でも触れる文脈（B）

### 3-4. 目標分布（参考値）

「非エンジニアが入口にいる」前提なので、A+B が **全体の 30〜40%** に達することを目標にします（現状 64/325 ≒ 20% は B が薄すぎる可能性が高い）。

| 値 | 目標シェア | 現状 |
|---|---|---|
| A | 5〜10% | 6%（20件） |
| B | 20〜30% | 14%（44件）← **要拡大** |
| C | 30〜35% | 33%（105件） |
| D | 25〜30% | 36%（116件） |
| E | 5〜15% | 11%（34件） |

ただし「数合わせで上下に振る」のは禁止。1件ずつ Q1〜Q4 で判定し、結果として比率が動く。

## 5. 判定の手順（チェックリスト）

新規執筆・既存見直しのとき、この順で決めます。

1. **`experience_level`** を §1 のフローで決める（事実問題、迷いが少ない）
2. **`importance`** を §3-2 の Q1〜Q4 で決める（読者基準、最初に決めると軸がブレない）
3. **`reader_level`** を §2-4 のフローで決める（本文の前提と整合）
4. §4 の組み合わせパターンと照合し、極端な組み合わせ（例：`importance: A` × `reader_level: 5`）になっていないか確認
5. 違和感があれば 2 と 3 を再考。`experience_level` は変えない

---

## 6. 既存値の見直しガイド

- **§5 の順で全件再判定**する。values が既に入っていても、現状を見ずに「もしゼロから決めるとしたら」で判定し直してから、現状と突き合わせる
- **動かす条件**: ポリシーで導出した値と現状が **2 段以上**離れているとき（`reader_level: 2 → 4` や `importance: A → C`）。1 段差は許容範囲として動かさない
- **大きく変える場合は備考に残す**: `備考` 節に「2026-05-27: ポリシー v1 で reader_level 4 → 2 に下方修正」のように一行入れる
- **status は触らない**。`ready` のままレベルだけ更新する

## 7. Validator との関係

| 軸 | 現在の validator | 本ポリシーで追加したい検査 |
|---|---|---|
| experience_level | enum 検査のみ（⚠️） | 本文の強い断定（「最強」等）と `research_only` の同時出現を警告 |
| reader_level | フォーマット検査（`\d` または `\d-\d`） | 範囲幅 2 以上（`"2-4"` 以上）を ⚠️、クォート無し `2-3` を ⚠️ |
| importance | enum 検査のみ（⚠️） | 全体分布が目標シェアから大きく外れたら集計レポートで表示（個別判定ではなく月次レポート） |

実装は本ポリシー確定後に [scripts/validate_entry.py](../scripts/validate_entry.py) に追加します。
