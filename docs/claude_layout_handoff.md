# Claude 作業引き渡しメモ — 本文見開きの実寸PDF調整

*作成日: 2026-05-14*

## 目的

Claude に本文エントリの調整を依頼するときの前提メモです。

いまの課題は、validator や `check_entry.py` 上は通っていても、PDF 実寸で見ると本文・右ページ要素が下で切れることです。特に B 章以降の通常エントリでは、A 章の前付けページより本文密度が高いため、**TypeScript 見本の密度を基準にして原稿量を落とす**必要があります。

## 重要な結論

- A 章は前付けなので、本文ページ密度の基準にしない。
- 基準は `drafts/prototypes/mockups/design_philosophy_v2/typescript_spread.html`。
- 「2 ページに収まった」だけでは OK ではない。下が切れていないことを目視で確認する。
- CSS で無理に押し込まない。先に原稿を短くする。
- `overflow: hidden` で切れを隠すのは禁止。切れるなら切れて見える状態で検出する。

## 参照ファイル

- 基準見本: `drafts/prototypes/mockups/design_philosophy_v2/typescript_spread.html`

## PDF 実寸の前提

本の片ページは次を基準にします。

- 片ページ: `199mm × 281mm`
- 見開き: `398mm × 281mm`
- 1 エントリ: 原則 2 ページ

PDF 出力時は、片ページ `199mm × 281mm` で確認します。画面プレビューの px サイズだけで判断しないでください。

## 現状の問題

次のような状態は NG です。

- PDF ページ数は 2 ページだが、下部の要素が切れている。
- 右ページの「開発フロー」「関連用語」「参考 URL」が見切れている。
- 左ページのメイン図・ポンチ絵まわりが本文に押されて潰れている。
- `overflow: hidden` により、切れが検出できない。
- 文章量を維持したままフォントサイズや余白だけで対応しようとしている。

## Claude に頼む作業方針

### 1. まず通常エントリで確認する

A 章ではなく、B 章以降の通常エントリで確認してください。

優先確認対象:

- `B-3 ChatGPT`
- `F-50 git`
- `J-14 LLM`
- `B-1 Gemini`
- `B-2 Claude`
- `G-1 Context`
- `I-1 MCP`

PDF 生成例:

```bash
python scripts/preview_to_pdf.py --only B-3,F-50,J-14
```

PDF ビューアで実際に開き、下部が切れていないか確認してください。

### 2. TypeScript 見本に近い密度へ削る

文章は「許容上限」ではなく、**誌面で自然に見える量**に寄せます。

実運用の目安:

| 箇所 | 目安 |
|---|---:|
| tagline | 30〜45 字 |
| 何をしてくれるか | 80〜120 字 |
| どこで出会うか | 80〜120 字 |
| 6 視点 | 各 15〜28 字 |
| 開発フロー | 4 ステップ優先 |
| 関連用語 | 3〜4 個優先 |
| ポンチ絵 caption | 40〜55 字 |
| 会話での使い方例 | 30〜40 字 |

`docs/v2_rules_summary.md` の許容値は「機械的に壊れていない上限」です。実際の組版では、上限いっぱいまで書かないでください。

### 3. 右ページを特に短くする

右ページは切れやすいです。

優先して削る場所:

1. 6 視点の長い文
2. 開発フローの補足
3. 関連用語の数
4. 参考 URL 周辺の長い注記

右ページでは、各セルを説明文にしすぎないでください。

悪い例:

```text
使い方を間違えると、期待した回答が返らず、原因の切り分けも難しくなります。
```

良い例:

```text
用途を外すと原因切り分けが難しくなります。
```

### 4. 左ページも図の余地を残す

左ページは本文だけで埋めないでください。メイン図・ポンチ絵が主役なので、本文は説明の入口にとどめます。

削る優先順位:

1. 似た説明の重複
2. サービス名や機能名の列挙
3. 歴史・背景の細部
4. 「つまり」「言い換えると」の再説明

### 5. CSS 調整は最後

原稿を短くしてもなお少しだけ苦しい場合のみ、CSS を触ってください。

許容される CSS 調整:

- 右ページのカード間 gap を少し縮める
- フローのステップ高を少し下げる
- 関連用語ピルの余白を少し下げる

避ける CSS 調整:

- 本文フォントを大幅に小さくする
- `overflow: hidden` で切る
- ページサイズを変える
- ページ下部のチロムを消す
- 画像や図を極端に小さくする

## 作業手順

1. 対象エントリを 1 件選ぶ。
2. `check_entry.py` で機械チェックする。
3. PDF を生成する。
4. PDF を開いて下部切れを目視する。
5. 切れていたら CSS ではなく Markdown 原稿を削る。
6. 再度 PDF を生成する。
7. TypeScript 見本と密度を比べる。
8. 問題なければ次のエントリへ進む。

## 最初に直す候補

まず `B-3 ChatGPT` を TypeScript 見本の密度に合わせてください。ここを基準稿にしてから、`B-1`, `B-2`, `F-50`, `J-14` に展開します。

`B-3` で確認すること:

- 左ページの下部までメイン図・ポンチ絵が見えるか
- 右ページの開発フローが切れていないか
- 関連用語と参考 URL が見えるか
- 6 視点が説明過多になっていないか
- PDF が 2 ページで、かつ見切れがないか

## 判定基準

OK:

- PDF が 2 ページ。
- 各ページ下部のチロムが見える。
- 右ページの関連用語・参考 URL まで見える。
- TypeScript 見本と同程度の余白・密度に見える。
- 本文を読まなくても、左ページの図とタグラインで用語の方向がわかる。

NG:

- PDF は 2 ページだが下が切れている。
- 右ページ下部が見えない。
- 文字が詰まりすぎて辞書ページに見えない。
- 図が本文に負けている。
- CSS で無理に縮めて読みにくくなっている。
