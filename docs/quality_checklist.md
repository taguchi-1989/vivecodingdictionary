# エントリ品質チェックリスト

*1 エントリを書き終えたら、このファイルを上から下へなぞって YES/NO で答えていくだけの、機械的な確認表です。*

- **対象**：`spread_v1` テンプレで書いた 1 エントリ
- **所要時間**：5 分以内を目標
- **関連ドキュメント**
  - 原則（なぜそのルールか）：[quality_guidelines.md](quality_guidelines.md)
  - トーン・文体の細則：[editorial_style.md](editorial_style.md)
  - 根っこの思想：[book_philosophy.md](book_philosophy.md)
  - 構造の正解：[../templates/entry_template.md](../templates/entry_template.md)

## 使い方

1. ドラフトが書き上がったら、上から順にチェックする
2. `☆` がついた項目は **ready 前に必ず YES** にする（致命的な逸脱）
3. 本文中に違反箇所があれば、そのまま修正 → 再スキャン
4. 全部 YES になったら `entries.csv` の `status` を `drafting` → `needs_review` に更新

## A. YAML フロントマター

- [ ] ☆ `id` が [id_scheme.md](id_scheme.md) の letter + 2 桁サブ範囲と一致している
- [ ] ☆ `title` が誌面に出る表記（カタカナ訳つきなら `Context（コンテキスト）` のように）になっている
- [ ] `category` と `subtype` が棚分けとして矛盾していない（サービス名とモデル名を混ぜていない）
- [ ] `experience_level` が `hands_on` / `partial` / `research_only` のいずれかで、本文の断定度と一致している
- [ ] `reader_level` が 1〜6（または "2-3" 形式）で、本文の深さと合っている
- [ ] ☆ `figure_type` が `before_after` / `structure` / `comparison` / `workflow` / `timeline` のいずれかで、メイン図の中身と一致
- [ ] `page_layout: spread_v1` が入っている
- [ ] ☆ 時変情報を扱うエントリに `evaluation_date` が入っている（YYYY-MM-DD）
- [ ] モデル・サービスには `version_status` / `pricing_note` が適切に入っている
- [ ] `related_terms` が 3〜6 個、本文末尾の「関連用語」と一致している
- [ ] ☆ `status` が `drafting` から始まっている（いきなり `ready` にしない）

## B. 左ページ

### tagline

- [ ] ☆ 25〜40 字に収まっている
- [ ] です・ます調
- [ ] カタカナ語が入る場合、読み仮名か日本語訳が添えられている（例：`git（ギット）`）

### ひとことで

- [ ] **50〜80 字**（1〜2 文）で収まっている
- [ ] 結論から書かれている（定義 → 補足の順）
- [ ] 専門用語をそのまま使っていない（初出で日本語訳か役割を補う）

### 何をしてくれるか

- [ ] **250〜350 字目安**（本文のみ、箇条書き部分も含む）
- [ ] 仕組みや役割が具体的（比喩だけで逃げていない）
- [ ] 難易度が `reader_level` と合っている
- [ ] 長いコードブロックで誌面を埋めていない

### バイブコーディングでの位置づけ

- [ ] **180〜280 字目安**で「いつ・どの工程で・どう関わるか」が書かれている
- [ ] 純粋な業務意味説明ではなく、**AI との打ち合わせ・指示・現場** の視点で書かれている
- [ ] 著者の体感（「〜ことがあります」）と公式仕様が同じ段落で混ざっていない
- [ ] 著者の個人的な気づき・エピソードは本文に入れず、**著者の「私のコメント」欄に回せる**ようになっている

### メイン図

- [ ] 「図の狙い」が 1〜2 文で書かれている
- [ ] `figure_type` に対応するサブブロック（A/B/C）のみ残り、使わないブロックは削除されている
- [ ] ☆ 図の設計が「人の視点」主役になっている（= ポンチ絵メモに登場人物・吹き出しが具体指示されている。[MEMORY](../../.claude/memory の該当メモ) 参照）

### 関連用語

- [ ] 3〜6 個
- [ ] `用語名 — 短いメモ` 形式で隣接関係が一言書かれている
- [ ] YAML `related_terms` と順序・表記が一致している
- [ ] 他エントリに ID がある場合、ID を併記している（例：`G-46 ナーフ`）

## C. 右ページ

### この用語の見どころ（6 視点）

- [ ] ☆ 6 つすべて埋まっている（1.役割 / 2.うれしさ / 3.注意点 / 4.どこで役立つか / 5.最初に理解する範囲 / 6.深掘り先）
- [ ] 各視点が **1 文・20〜40 字目安**に圧縮されている（6 視点合計 150〜240 字）
- [ ] 6.深掘り先だけは語のカンマ区切りでも可（例：`rebase、cherry-pick、worktree`）
- [ ] 「注意点」は誤解・副作用・学習コストのいずれか具体（「特にありません」系は NG）

### 開発フローでの位置（必須）

- [ ] ☆ 4〜5 ステップある（既存サンプルは全件 4 ステップ）
- [ ] 各ステップが「短ラベル — 1 行補足」形式（1 ステップ 30〜40 字目安）
- [ ] ☆ サービス／モデル／歴史エントリでも「その名前が実際の開発のどこで呼ばれるか」が書かれている（省略不可）

## D. 著者記入欄（AI は触らない）

- [ ] ☆ `非エンジニア視点のつまずき` は `-\n-\n-` の空箇条書きのまま
- [ ] ☆ `私のコメント` は `🙂 第一印象: ` `👍 良い点: ` `👎 ダメな点: ` `🎯 誰に向くか: ` の 4 行スケルトンが空のまま
- [ ] ☆ AI が気を利かせて中身を埋めていない（埋めた場合は必ず削除）

## E. 裏台帳メモ

### 誌面ポンチ絵メモ

- [ ] メイン図欄に「描く内容」「登場人物」「吹き出し・心の声」「中央キーワード」が埋まっている
- [ ] ☆ 登場人物と吹き出しが具体（抽象図のみは NG。「人の視点」方針に準拠）
- [ ] 6 視点アイコン欄は「共通アイコン流用」か、個別演出が要るときだけ記述
- [ ] 開発フロー図欄に各ステップのアイコン／絵柄が書かれている

### 出典メモ

- [ ] ☆ 時変情報（モデル、料金、提供状況、リリース日）を扱う場合、出典が `- ソース名 — checked YYYY-MM-DD` 形式で入っている
- [ ] 公式ドキュメントか一次情報を参照している（二次記事だけの場合は `needs_source` のまま止める）

### コミュニティ補完メモ・備考

- [ ] 近接エントリとのスコープ分担（どこまでこのエントリで扱い、どこから別エントリか）が書かれている
- [ ] 時変情報の再確認タイミング、例外ケース、別サービスとの区別が必要なら備考にメモされている

## F. トーン・文体（横断チェック）

- [ ] ☆ 本文全体が「です・ます」調（「〜だ」「〜である」が混ざっていない）
- [ ] ☆ カタカナ語が初出で日本語訳・役割が補われている（例：`Context（コンテキスト） ＝ 文脈`）
- [ ] ☆ 略称が初出で展開されている（例：`Claude Code（略称 CC）`）
- [ ] 強い断定が混ざっていない（「最新」「最強」「完全」「絶対」「必ず〜できる」など）
- [ ] 上から目線が混ざっていない（「初心者でも分かる」「簡単です」「誰でも〜」など）
- [ ] ベンチマーク数値だけで優劣を決めていない
- [ ] 断定できない事象は「ことがあります」「場合があります」で弱化されている
- [ ] 太字 `**...**` は概念のキーワードに限定されている（装飾的な強調になっていない）
- [ ] 一文が長すぎない（句点で読点 3 つを超える文は要分割検討）

## G. スコープ整合

- [ ] 隣接エントリ（`related_terms` に出てくる ID）と内容が重複していない
- [ ] 時期差分（例：Claude 3.5 系と Claude 4 系）は別 ID で分けてあり、同エントリで混ざっていない
- [ ] 会社（`person` カテゴリ）とサービス（`service`）とモデル（`model`）が同エントリに混在していない
- [ ] 料金・モデル名は専用エントリに逃がして、本文では簡潔に触れるだけにしている

## H. 誌面量の目安

*誌面デザインはメイン図に左ページの 40% 前後を使う想定です。本文が詰まりすぎないように、各セクションと合計で目安に収めます。*

### 左ページ本文（誌面に出る量）

| セクション | 目安 |
| :-- | --: |
| tagline | 25〜40 字 |
| ひとことで | 50〜80 字 |
| 何をしてくれるか | 250〜350 字 |
| バイブコーディングでの位置づけ | 180〜280 字 |
| 関連用語（6 × 短ラベル） | 100〜150 字 |
| **左ページ合計（本文のみ）** | **700〜800 字** |

### 右ページ本文（誌面に出る量）

| セクション | 目安 |
| :-- | --: |
| 見どころ 6 視点（各 20〜40 字） | 150〜240 字 |
| 開発フロー 4〜5 ステップ（各 30〜40 字） | 120〜200 字 |
| **右ページ合計（本文のみ）** | **250〜400 字** |

### チェック項目

- [ ] ☆ 左ページ本文合計が **700〜800 字** に収まっている（tagline + ひとことで + 何をしてくれるか + バイブ位置づけ + 関連用語）
- [ ] ☆ 右ページ本文合計が **250〜400 字** に収まっている（見どころ + 開発フロー）
- [ ] 裏台帳メモ（誌面ポンチ絵メモ ／ コミュニティ補完メモ ／ 出典メモ ／ 備考）は誌面に出ないので字数制限の対象外

### 超過したときの削り方（優先順）

1. **見どころ／備考と重複している記述**：本文側を削り、見どころ側に一文で残す
2. **著者の個人的な気づき・エピソード**：本文から外して「私のコメント」「備考」へ移す
3. **他エントリ（related_terms にある ID）の領域に踏み込んでいる記述**：該当エントリへ誘導して短くする
4. **長いコードブロック・箇条書き**：3〜4 行に圧縮、または図に寄せる

## 不合格時の戻し方

| 不合格項目 | 戻すステータス |
| :-- | :-- |
| A.YAML の欠落、D.著者欄への AI 記入 | ただちに `drafting` で修正 |
| E.出典メモの欠落・弱さ | `needs_source` に落とす |
| F.トーン・文体の逸脱 | `drafting` で修正（☆ が 3 個以上 NO なら全面書き直し検討） |
| B/C 構造の軽微な欠損 | `drafting` で修正 |
| H. 誌面量の超過 | 超過 100 字までは `needs_review` で残す。超過 100 字超は `drafting` に戻して削る |
| その他の軽い詰め | `needs_review` のまま、レビュー時コメント |
