# 編集スタイルガイド

*本書の根っこのフィロソフィーは [book_philosophy.md](book_philosophy.md) を参照。本ファイルは日々のトーン・文体の運用ルールに絞る。*

## 基本トーン

- 難しすぎず、しかし幼稚にしない
- 読者を馬鹿にしない
- 結論から書く
- 専門用語を消すのではなく、足場をつくる
- 事実と主観を混ぜない

## 本文の基本順（spread_v1、v2 誌面）

### 左ページ

1. tagline（25〜40 字、1〜2 文で用語を言い切る）
2. 何をしてくれるか（60〜100 字）
3. どこで出会うか（70〜110 字、旧「バイブコーディングでの位置づけ」）
4. メイン図（Before/After ／ 登場シーン ／ 概念図 のいずれか）
5. 関連用語（3〜5 個、誌面ではタグ表示）

### 右ページ

1. この用語の見どころ（6 視点、各 20〜30 字）
   - 1.役割 / 2.うれしさ / 3.注意点 / 4.どこで役立つか / 5.はじめに / 6.深掘り先
2. 非エンジニアのつまずき（3〜4 項目、誌面印刷、著者本人のみ記入）
3. 私のコメント（4 ラベル、誌面印刷、著者本人のみ記入）
4. 開発フローでの位置（必須、4〜5 ステップ）

### 裏台帳メモ（誌面には出さない）

1. 誌面ポンチ絵メモ
2. コミュニティ補完メモ
3. 出典メモ
4. 備考

*変更点（v2、2026-04-24）：「ひとことで」を廃止（tagline に合流）、「バイブコーディングでの位置づけ」→「どこで出会うか」、「最初に理解する範囲」→「はじめに」、著者欄を裏台帳から右ページ下段の誌面構成要素に昇格、「私のコメント」の 4 つ目の絵文字を 🎯 → 👥。*

## 文体

- **「です・ます」調で書く**（読者と近い距離を保つ）
- 一文を長くしすぎない
- 「つまり」「たとえば」を使って橋をかける
- カタカナ語は初出で日本語訳・役割を補う（例：Context ＝ 文脈）
- 断定できないものは「ことがあります」「場合があります」と書く
- 比喩は便利だが、仕様説明の代わりにしない

## 避ける書き方

- 「初心者でもわかる」のような上から目線
- 最新、最強、完全、絶対などの強い断定
- ベンチマークだけで優劣を決める説明
- 公式仕様と自分の体感を同じ段落で混ぜる説明
- 長いコードで誌面を埋める説明

## 用語の扱い

- サービス名、モデル名、会社名、言語名を混ぜない
- `category` と `subtype` で棚を分ける
- 読者が辞書引きできるように関連語を 3〜5 つ残す
- 正式名称と会話での呼ばれ方が違う場合は両方書く
- 初出の略称は必ず展開する（例：CC ＝ Claude Code）

## 略称・ヌメロニムの扱い（2026-04-28 追加）

タイトルが略称・略字・ヌメロニム（MCP / LLM / a11y / i18n / GDPR など）の場合は次のルールを守ります。

- **`title`** はそのまま（`MCP` / `a11y`）。括弧書き読みは付けない
- **`title_reading`** には**英単語をカタカナで並べた読み**を入れる（例: `モデルコンテキストプロトコル`（MCP）／`ラージランゲージモデル`（LLM）／`アクセシビリティ`（a11y））。略称音読み（えむしーぴー 等）や日本語訳（大規模言語モデル 等）はここに入れない
- **`tagline`** の冒頭に正式名称または日本語展開を 1 句で入れる。形は **「{展開} の略。{定義 1 文}。」** が基準
  - 英字略称：`Model Context Protocol の略。LLM とツール・データをつなぐ標準規格です。`
  - ヌメロニム：`accessibility の略（a と y の間に 11 文字）。Web を誰でも使える状態に保つ取り組みです。`
  - 日本語展開：`大規模言語モデルの略。文章で訓練された AI の主流で、主要 3 サービスの中身です。`
- tagline の許容字数は 25〜60、推奨 35〜50（略称展開ぶん上限を緩めています）。詳細は [v2_rules_summary.md §2-3](v2_rules_summary.md) と [quality_checklist.md §H](quality_checklist.md) を参照
