# モバイル・コメント投入フロー 要件定義

*外出先のスマホ（Android）で著者コメント・つまずき・AI 生成部分への指摘を書き溜め、PC で落ち着いて本編 Markdown に取り込むための仕組みの要件定義です。v0.1（2026-04-24）。*

## 1. 目的と背景

### 目的

著者が PC 前にいない時間（移動中・通勤中・待ち時間）を、エントリの **著者メモを溜める時間** に変えることです。335 件を書き切るまでのスループットを底上げする狙いです。

### 背景

- テンプレ `spread_v1` の **著者記入欄**（非エンジニアのつまずき／私のコメント）は、AI が触らないため、著者が自力で埋めないと永遠に空
- 各エントリを実アプリ・実携帯で触ってから感想を書くのが理想だが、PC 前でまとめて書くのはハードルが高い
- 加えて、AI が書いた本文節に対しても「この節の説明は古い／ここは違う」の指摘を外で溜めておきたい

### スコープ

- スマホで 3 種類のコメント（詳細は §3）を溜める仕組み
- PC で Claude Code に読み込ませ、対応エントリに反映する仕組み
- **スコープ外**：本文自体の執筆（これは PC + [skills/write-entry.md](../skills/write-entry.md) で行う）／誌面デザイン

## 2. 前提条件

- スマホは **Android**
- モバイル側 UI は **Obsidian Mobile** を既製アプリとして流用する（自作アプリは作らない）
- 同期とストレージは **GitHub 1 本** に集約する
- 認証は著者個人のみ（他者のインプットは将来拡張）

## 3. 対象コメント 3 種

| kind | 反映先 | 既存テンプレ上の受け口 |
| :-- | :-- | :-- |
| `my_comment` | 右ページ「私のコメント」の 4 項目 | あり（🙂 / 👍 / 👎 / 🎯） |
| `stumble` | 右ページ「非エンジニアのつまずき」の箇条書き | あり |
| `ai_feedback` | AI が書いた本文節への指摘メモ | **無い → 裏台帳「備考」に新設追記** |

### `ai_feedback` の扱い（重要）

- **本文を直接書き換えない**。外出先でノーチェックで打った指摘が、そのまま本文に反映されるのは危険
- 指摘は裏台帳の「備考」欄の末尾に `> [著者指摘 YYYY-MM-DD / 対象節: X] コメント` として積む
- 後日、PC で著者と AI が一緒に「この指摘たちを本文にどう反映するか」を別タスクとして行う（2 段構え）

## 4. 全体データフロー

```
[スマホ / Obsidian Mobile]
    ↓ Obsidian Git プラグインが自動 push
[GitHub / このリポジトリの mobile_inbox/ 配下]
    ↓ PC で git pull
[Claude Code / /import-comments スキル]
    ↓ パース → 反映
[content/entries/**/*.md の著者欄・備考欄]
    ↓ 処理済み行は mobile_inbox/processed/ に移動
```

GitHub 1 本で完結するため、別サーバ・別サービスは使いません。

## 5. スマホ側（Obsidian Mobile）

### セットアップ

- Android に **Obsidian Mobile** をインストール
- このリポジトリ（`VibeCodingDictionary`）を vault として開く
- コミュニティプラグイン **Obsidian Git** を有効化し、以下を設定：
  - Auto pull interval：5〜15 分
  - Auto commit & push：有効（インターバル 5〜15 分）
  - 認証：GitHub Fine-grained PAT（`contents: write` 権限）

### 書き込み先

- ファイル：`mobile_inbox/YYYY-MM-DD.md`（日付ごと 1 ファイル）
- 存在しない日付のファイルは新規作成。存在する日はセクション追記

### 見出し規約（パース対象）

1 コメント ＝ 1 セクション（H2 見出し 1 個＋本文）。見出しのフォーマットは以下で固定します。

```markdown
## <entry_id> / <kind> [/ <sub>]

<本文テキスト>
```

実例：

```markdown
## B-2 / my_comment / first_impression
初見シンプル。でも3つの入口があると知って最初は戸惑った。

## B-2 / stumble
Opus/Sonnet/Haikuの3段階が、値段と賢さのどちらで分かれているのか最初わからなかった。

## B-2 / ai_feedback / 何をしてくれるか
4.6時代にラインナップ更新されたので書き直し候補。

## F-50 / my_comment / good
差分が取れる、の説明がしっくりきた。
```

### `sub` の意味（kind ごと）

| kind | sub の取りうる値 | 意味 |
| :-- | :-- | :-- |
| `my_comment` | `first_impression` / `good` / `bad` / `target` | 🙂 / 👍 / 👎 / 🎯 の 4 項目に対応 |
| `stumble` | （なし／省略） | 1 箇条として末尾に追記 |
| `ai_feedback` | 指摘対象の節名（日本語可、例：`何をしてくれるか`） | どの節への指摘かの手がかり |

### 入力補助（任意）

- Obsidian の **Templates** コアプラグインで、上記フォーマットの空セクションを 1 タップ挿入できるようにする
- テンプレは `templates/mobile_inbox_section.md` として vault 内に置く（別途）

## 6. リポジトリ側の規約

### ディレクトリ構成

```
mobile_inbox/
├── 2026-04-24.md          # その日の受信箱
├── 2026-04-25.md
└── processed/             # /import-comments 処理済み
    ├── 2026-04-23.md
    └── ...
```

- `mobile_inbox/` 直下が **未処理** キュー
- `mobile_inbox/processed/` が **処理済み**
- 取り込みが終わったファイル単位で `processed/` へ移動

### .gitignore / 扱い

- `mobile_inbox/` は **Git 追跡対象**（PC 側と同期のため）
- スマホ側は Obsidian Git が自動コミットするので、1 日に複数コミットが積まれる前提
- 履歴が気になる場合は定期的に squash するか、`mobile_inbox/processed/` を別ブランチで管理

## 7. PC 側（`/import-comments` スキル）

v0.1 では仕様のみ定義し、実装は別タスクで行います（§9 実装段階を参照）。

### 入力

- `mobile_inbox/*.md`（`processed/` 以外）のすべて

### 処理

各ファイルを上から走査し、H2 見出しと本文のペアを 1 コメントとして解釈：

| kind | 反映先の具体操作 |
| :-- | :-- |
| `my_comment` | `content/entries/**/{entry_id}_*.md` を特定 → 右ページ「私のコメント」の該当ラベル行（🙂 / 👍 / 👎 / 🎯）の後ろに本文を **追記**（既存テキストがあれば改行＋追記） |
| `stumble` | 同エントリ「非エンジニアのつまずき」の空 `-` 行を 1 つ埋める。全埋まりなら行を追加 |
| `ai_feedback` | 同エントリ「備考」欄の末尾に以下の 1 行を追加：`> [著者指摘 YYYY-MM-DD / 対象節: {sub}] {本文}` |

### エラー時

- `entry_id` が `ledgers/entries.csv` に存在しない → そのコメントだけスキップし、ログに記録
- `kind` 不明 → 同上
- 本文が空 → 同上
- エラーがあっても他のコメントの処理は続行する

### 完了時

- 全コメントが正常反映されたファイルは `mobile_inbox/processed/` へ移動
- 部分失敗したファイルは元の場所に残し、エラーコメントだけ対象外とする仕組みにする（具体方式は実装時決定）
- `ledgers/entries.csv` の対象エントリの `status` を、必要に応じて `needs_review` に更新（著者欄が埋まったので品質チェックに回す意図）

### 呼び出し方

- Claude Code で手動：「import-comments 実行して」の自然言語指示
- 実体は `skills/import-comments.md`（今後作成）

## 8. セキュリティと運用

### PAT の扱い

- Fine-grained PAT を使い、**このリポジトリだけ**に権限を絞る
- 有効期限は 90 日目安（Obsidian Git が失効時に警告してくれる）
- PAT は Android の Obsidian 内だけに保存。GitHub 本体の SMS 2FA は別途維持

### 機密情報

- `mobile_inbox/` に機密情報は書かない前提（人事情報、パスワード等は対象外）
- 万一書いた場合は git history ごと修正する運用（スコープ外）

### 競合

- PC 側で編集中に Obsidian Git が push → merge conflict の可能性
- 対策：Obsidian Git の pull 間隔を先に設定し、PC 側で作業する前に必ず `git pull`。ローカルで `mobile_inbox/` を触らないことを自己ルール化

## 9. 段階的実装計画

最小実装から始めて、回る形にしてから拡張します。

### Phase 1：素の運用（プラグイン設定のみ）

1. Android に Obsidian Mobile と Obsidian Git を入れ、vault を開く
2. 適当に `mobile_inbox/2026-04-24.md` を手動で作り、見出し規約で 1〜2 件書いてみる
3. 自動 push が動くか確認
4. PC 側で `git pull` してファイルが来ているか確認

ここまでで「外で書き溜められる」状態になります。

### Phase 2：`/import-comments` スキルの実装

1. `skills/import-comments.md` を新規作成（手順書）
2. 実際の反映操作は Claude Code に自然言語で依頼する形（初期は完全なスクリプト化はしない）
3. 1〜2 回手動で回して、取り込み精度と運用感を確かめる

### Phase 3：自動化・補助

- Obsidian Templates でセクション挿入テンプレを整備
- `/import-comments` の処理を Python / Node スクリプト化して誤取り込みを減らす
- エラー時の通知方法（issue 化など）の整備

## 10. 今後の拡張余地

- **音声入力**：Android のキーボード音声で入力できるので、特別な対応は不要。ただし誤変換が多ければ、テンプレ側で `kind` が分かりやすい表記に緩める
- **新規エントリの起票**：外出先で「これエントリ化したい」と思った語を、`mobile_inbox/new_candidates.md` に書き溜める運用を追加できる
- **本文の改訂モード**：Phase 2 以降で、`ai_feedback` を AI に再度読ませて本文改訂案を PR として作る運用（現段階はスコープ外）
- **他者入力**：編集者・家族など別の入力者が出てきたら、GitHub の別 PAT／別ブランチで分ける
