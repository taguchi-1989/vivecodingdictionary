# ゲラ・入稿運用プラン

*2026-06-13。「コンテンツの作り込みに詰まってきたので、いちど本として通して見たい」という段階に入ったための運用手引きです。どこで・どんな形式で刷ってゲラとして見ていくかを、目的別に整理しています。*

*いまの最優先は **ルート A（内容・流れの通読校正）**。ルート B / C は内容が固まってからの段階なので、ここでは全体像と判断ポイントを押さえる程度にとどめます。*

関連：[docs/production_plan.md](production_plan.md)（制作計画）／ [docs/v2_rules_summary.md](v2_rules_summary.md) §7（書籍化に残る課題）／ [book_readiness_review.md](../drafts/prototypes/mockups/design_philosophy_v2/book_readiness_review.md)（印刷工程レビュー）。

---

## 0. 結論（先に読む）

- ゲラは **A → B → C の 3 段階**で考えます。いきなり印刷所に入稿しません。
- **いま作るべきは「1 冊ぶんを通し順で束ねた PDF（簡易ゲラ）」**。自宅プリンタかコンビニのネットプリントで A4 に束で刷り、赤ペンで通読校正します。
- **塗り足し（bleed）・CMYK 色校・ノド非対称マージン**は CSS 実装担当のスコープです（[v2_rules_summary.md](v2_rules_summary.md) §7）。ルート A はそれらが未整備でも回せます。ルート B / C はそれらが入ってから。
- 判型は **150mm × 212mm 縦**（ISO A 系 √2 準拠）で確定済み。`preview_to_pdf.py` がこのサイズで PDF を出します。

## 1. ゲラの 3 段階

| ルート | 何を確かめる | どこで刷る | 形式 | 1 回コスト目安 | いま可能か |
|---|---|---|---|---|---|
| **A. 簡易ゲラ** | 文字・構成・抜け漏れ・通読のリズム | 自宅プリンタ／コンビニ（ネットプリント） | A4 に PDF（実寸 or 2up）、白黒可 | 数百〜2,000 円 | **今すぐ可** |
| **B. 束見本ゲラ** | 本という"モノ"（厚み・見開きの収まり・手触り・重さ） | POD 製本サービス（1 部から） | 実寸 150×212mm で製本、PDF 入稿 | 1 部 1,000〜3,000 円＋送料 | bleed 等の整備後 |
| **C. 入稿ゲラ／色校** | 本刷り前の最終確認（色・トンボ・面付け） | 商業印刷所 | PDF/X-1a・CMYK・トンボ・塗り足し | 数千〜数万円 | 実装担当の整備が前提 |

**進め方の原則**：A で内容を固める → B で 1 部だけ現物を取って体裁を確かめる → 不満点を反映 → C で本刷り。A を飛ばして B/C に行くと、紙とインク代で校正することになり高くつきます。

## 2. ルート A：簡易ゲラ（内容・流れの通読校正）★いまここ

### 2-1. 何のためにやるか

339 件を画面で 1 件ずつ見ていると「本として通したときの流れ」が見えません。束で刷って手でめくると、以下が一気に分かります。

- 章の入り・終わりのリズム、同型レイアウトの単調さ（[book_readiness_review.md](../drafts/prototypes/mockups/design_philosophy_v2/book_readiness_review.md) §2 の課題）
- tagline の言い回しのダブり、隣り合うエントリの説明の重複
- 抜けている語・順番の違和感（`entries.csv` 順が読み物として自然か）
- 1 見開きに収まっていないエントリ（`preview_to_pdf.py` のはみ出し判定で機械検出済みのものを、実紙で確認）

### 2-2. 通し PDF の作り方

通し順の正本は [ledgers/entries.csv](../ledgers/entries.csv)（356 行）の並び。前付けは [drafts/front_section/](../drafts/front_section/)（概念見開き〜凡例〜ログ/用語集の 7 種）、後付けは [drafts/back_section/afterword.html](../drafts/back_section/afterword.html)。

現状の生成資産：

- [scripts/preview_gen.py](../scripts/preview_gen.py)：md → 見開き HTML（`drafts/prototypes/preview/` に出力、prev/next 付き）
- [scripts/preview_to_pdf.py](../scripts/preview_to_pdf.py)：各 HTML を **150×212mm の PDF** に変換し、2 ページに収まっていないものを警告

**いまある手段（暫定）**：

1. `python scripts/preview_gen.py` で preview HTML を生成
2. `python scripts/preview_to_pdf.py` で **エントリ単位の PDF** を `drafts/prototypes/preview/pdf/` に出力
3. それを 1 本にまとめる（後述の「次の一手」のスクリプトが入るまでは、PDF 結合ツールや印刷ダイアログ側で手動結合）

**次の一手（未着手・スクリプト化の対象）**：`entries.csv` 順に「前付け → A〜J 全エントリ → 後付け」を 1 本の PDF に束ね、ノンブル（ページ番号）を振った**簡易ゲラ PDF を 1 コマンドで出す**ビルドスクリプト。仕様は §5 に置きました。今回はドキュメント整備のみのため未実装です。

### 2-3. どこで・どんな形式で刷るか

| | 自宅プリンタ | コンビニ ネットプリント |
|---|---|---|
| 向くケース | 章ごと・差分だけ刷り直す日常運用 | 全 339 件を一気に束で刷る |
| 形式 | A4 / B5 に**実寸（150×212）配置**、または **2 見開き = 4 ページを A4 1 枚に 2up** | 同左。PDF をアップロードして店頭出力 |
| 色 | 白黒で十分（色校はルート C） | 白黒（1 枚 ¥10 前後）／カラー（¥50〜80） |
| 製本 | しない（クリップ／2 穴ファイル） | しない |
| 体裁の正しさ | △（プリンタ余白が出る） | ○ |

**推奨設定**：

- 用紙：A4。**「実際のサイズ」で印刷**（プリンタ側で「ページに合わせる／拡大縮小」をオフ）。150×212 の枠が A4 内に実寸で出るので、本のサイズ感が分かります。
- 面付け：通読校正なら **1 見開き = A4 横 1 枚**が読みやすい（赤入れ余白も取れる）。紙を節約するなら 2up。
- 両面：**片面推奨**（裏写りせず赤入れしやすい）。束見本的に厚みを見たいときだけ両面。
- ノンブル：通し PDF にページ番号が入っていると赤入れの指示出しが楽（§5 のスクリプトで対応予定）。

### 2-4. 赤入れ → 反映の戻し方

紙のゲラに入れた赤を、リポジトリに戻す経路は既にあります。

- スマホ／外出先：[docs/mobile_repoedit_setup.md](mobile_repoedit_setup.md)（RepoEdit で `user-input` ブロックを直接編集、戻り先 `mobile-drafts` ブランチ）
- PC：該当エントリを直接編集 or `entry-writer` サブエージェント呼び出し → 保存で [validate_entry.py](../scripts/validate_entry.py) が自動チェック → [revision_queue.md](../ledgers/revision_queue.md) が再生成
- 校正の優先順位は `cat ledgers/revision_queue.md` で確認（☆違反 / 警告 / 書きかけ / 著者レビュー待ち / 完成の内訳）

赤入れで「事実」を直すのは本文節、「主観」は著者記入欄のみ（AI は触らない）。この切り分けは [CLAUDE.md](../CLAUDE.md) §3 のとおり。

## 3. ルート B：束見本ゲラ（本という現物の確認）

内容がルート A で一段落したら、**1 部だけ製本した現物**を取って、画面では分からない「厚み・束（つか）・見開きの収まり・紙の手触り・重さ」を確かめます。

### 3-1. どこで（POD / 小ロット製本サービス・1 部から）

いずれも **PDF 入稿・1 部から・無線綴じ対応**が基本。仕様と価格は変動するため発注前に各社サイトで要確認です。

| サービス | 特徴 | 向く用途 |
|---|---|---|
| ちょ古っ都製本工房 | 1 冊から安価、ページ数の上限が大きい | 339 件＝厚めの本の束見本に向く |
| 製本直送.com | PDF アップロードで 1 冊から、操作が簡単 | 手早く 1 部だけ現物がほしい |
| グラフィック | 品質・紙種が豊富、オンデマンド〜オフセット両対応 | B で品質も見て C に橋渡し |
| しまや出版 ほか同人系 | 同人誌印刷のノウハウ、装丁の選択肢 | 表紙・装丁も試したいとき |

> 上記は代表例です。「オンデマンド印刷 1 冊 無線綴じ」で各社の最新仕様（最大ページ数・対応サイズ・PDF 要件）を確認してください。150×212mm は標準サイズでないことがあるので、**近い標準サイズ（A5＝148×210mm）に寄せるか、変形サイズ対応の業者を選ぶ**判断が要ります。

### 3-2. どんな形式で渡すか

- 形式：**PDF**（単ページ連続。見開きスプレッドではなく 1 ページ = 1 ページで出す業者が多い。要確認）
- サイズ：実寸 150×212mm（または A5 に寄せる）
- 塗り足し：紙端まで色や罫が来るなら 3mm の bleed が必要（[book_readiness_review.md](../drafts/prototypes/mockups/design_philosophy_v2/book_readiness_review.md) §1）。CSS 実装担当の対応待ち
- フォント：Zen Maru Gothic を**サブセット埋め込み**（SIL OFL＝埋め込み可）
- 色：オンデマンドなら RGB 入稿可の業者もあるが、原則 CMYK 想定で青（`#2F7BE6`）の沈みを覚悟。色校は本来ルート C

## 4. ルート C：入稿ゲラ／色校（本刷り前）

商業印刷の本刷りに向けた最終ゲラ。**CSS 実装担当が塗り足し・ノド・CMYK を整えた後**の段階です。ここでの主担当は実装担当＋印刷所で、本リポジトリ（markdown / 検証）の範囲外。

判断ポイントだけ控えておきます（詳細仕様は印刷所の入稿ガイドに従う）：

- 形式：**PDF/X-1a**（または印刷所指定）。トンボ・塗り足し 3mm 込み
- 色：**CMYK 変換 → 色校（本紙校正 or 簡易校正）必須**。青の彩度低下を実紙で確認
- 面付け：印刷所側で行うのが通常（こちらは単ページ連続入稿）
- フォント：完全埋め込み or アウトライン化
- チェック：[v2_rules_summary.md](v2_rules_summary.md) §7 の未解決チェックボックス（bleed / CMYK / フォント埋め込み / 章扉）が全部 ✅ になっていること

ルート A を回しやすくするため、後で実装する想定のスクリプト仕様。今回は未実装です（「実装して」と言われたら着手）。

## 7. 校正・入稿チェックリスト

### ルート A（通読校正）で見る
- ☐ 章の入り／終わりのリズム、同型レイアウトの単調さ
- ☐ tagline・説明のダブり（隣接エントリ間）
- ☐ 抜けている語、`entries.csv` 順の違和感
- ☐ 1 見開きに収まらないエントリ（`preview_to_pdf.py` 警告と実紙の突き合わせ）
- ☐ 略称・ヌメロニムの正式名称が tagline 冒頭に入っているか（[CLAUDE.md](../CLAUDE.md) §3）
- ☐ ですます調・事実/主観の切り分け

### ルート B/C（印刷工程）で見る ※実装担当と共同
- ☐ 縦横比 √2（750×1061）
- ☐ bleed 3mm
- ☐ ノド非対称マージン
- ☐ CMYK 色校（特に青 `#2F7BE6`）
- ☐ フォント埋め込み（Zen Maru Gothic サブセット）
- ☐ 章扉・章インデックス（フォアエッジ帯）
- ☐ 解像度 300dpi（画像素材を入れる場合）
