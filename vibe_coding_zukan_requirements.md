# バイブコーディング図鑑 企画・要件定義書（たたき台）

## 1. 企画概要

### 1-1. 仮タイトル
- バイブコーディング図鑑
- 非エンジニアのためのAI開発ことば図鑑
- 触って学ぶAI開発用語図鑑

### 1-2. 企画の一文要約
非エンジニアや周辺職種の人が、AI開発・フロントエンド開発・LLM活用の現場で飛び交う言葉を、図解と短文と体験メモで理解できるようにするための図鑑型書籍・配布資料。

### 1-3. 目的
- 自分が学びながら触ってきた知識を体系化する
- 現場で言葉が通じなくなる問題を埋める
- 勉強会・会話・打ち合わせの前提知識をそろえる
- 非エンジニアが最低限の語彙を辞書引きできるようにする
- 自分の実体験と、未体験だが調査済みの内容を区別して蓄積する
- 必要に応じてコミュニティ知見で補完できる土台をつくる

### 1-4. 想定読者
- AI開発やアプリ開発に関わり始めた非エンジニア
- 企画、事業、デザイン、運用、マネジメント職
- エンジニアと話す必要があるが、語彙の差で会話が詰まる人
- 社内勉強会の参加者
- 将来的には、書籍・配布冊子・社内資料の読者

### 1-5. 提供価値
- 難しい概念を図解で把握できる
- 単なる定義ではなく「何のための言葉か」がわかる
- 実務でどう出会うかがわかる
- 体験済み/未体験/調査ベースが明示されており、信頼の置き方がわかる
- 関連語へ辿れるため、語彙の地図として使える

---

## 2. 制作の背景・課題認識

### 2-1. 背景
- 自分では当たり前に使っている言葉が、相手には通じない場面が増えている
- たとえば「コンテキスト」「ディクテーション」などが共有語彙になっていない
- 勉強会や座談会でかなりレベルを落として話しても、それでも難しいと言われる
- 単なるハウツーに落とすと、自分が伝えたい感動や哲学や原体験が抜け落ちる
- 一方で、原理から全部説明すると長すぎて会話も学習も成立しない

### 2-2. 解決したい問題
- 語彙の非対称性が大きく、会話の前提が合わない
- その都度説明しても、場当たり的で蓄積しない
- 現場で最低限共有すべき概念セットがない
- 自分が何を当然視しているかを明文化できていない
- 体系だった入り口資料がないため、学びが断片化する

### 2-3. 本企画の役割
- 共有語彙の最小セットを提供する
- 「辞書引き」できる状態をつくる
- 会話の前提をそろえるための橋渡し資料になる
- 実体験を伴った言葉の説明を残す
- 自分の学びの棚卸しと再利用を可能にする

---

## 3. コンセプト

### 3-1. 基本コンセプト
「非エンジニアがAI開発まわりの言葉に追いつくための、図鑑形式の橋渡し本」

### 3-2. 差別化ポイント
- 図鑑形式である
- 実体験が入る
- 体験していないものは未体験・調査ベースと明示する
- コミュニティ補完の余地を前提に設計する
- 単なる定義集ではなく、現場で何の役に立つかを示す
- 「哲学」「原体験」「なぜこれが重要に見えるのか」も書き残せる

### 3-3. トーン
- 難しすぎず、しかし幼稚にしない
- 読者を馬鹿にしない
- かみ砕くが、本質を削りすぎない
- 事実と主観を分離する
- 余計な色を使わず、白・青系で整理する

### 6-4. 型D：歴史・系譜ページ

## 10. ID・番号設計

### 10-1. 基本方針
IDは「ページ番号」ではなく「管理番号」として付与する。
後から掲載順やページ順が変わっても、管理上の参照が崩れない構造を優先する。

### 10-2. 推奨方針
- 大分類ごとにレンジを予約する
- その中で連番を振る
- 将来の追加を見越して余白を残す
- 表示上のページ番号とは切り離す

### 10-3. 予約レンジ案
- 000-099：共通・序文・方針
- 100-199：Service（サービス）
- 200-299：Model（モデル）
- 300-399：Term（技術用語）
- 400-499：Tool（ツール）
- 500-599：Workflow（開発手法・進め方）
- 600-699：History（歴史・系譜）
- 700-799：Person（人物・会社）
- 800-899：Benchmark（ベンチマーク）
- 900-999：保留・未整理・実験枠

### 10-4. ID例
- 101：Gemini
- 201：Gemini 2.5 Pro
- 202：Gemini 2.5 Flash
- 203：Gemini 2.5 Flash-Lite
- 301：JavaScript
- 302：TypeScript
- 303：ESLint
- 601：Bard から Gemini への流れ
- 801：SWE-Bench

### 10-5. 詳細ルール
- 欠番は許容する
- 無理に詰めない
- 近い項目は近い番号帯に寄せる
- ただし、あとから無理に全面振り直しはしない
- 「整理のための番号」であり、「意味を詰め込みすぎた番号」にしない

### 10-6. 拡張ルール案
必要なら以下のようなサブIDを持てる。
- 201-a：Gemini 2.5 Pro の比較メモ
- 201-b：Gemini 2.5 Pro の時期差分メモ
- 601-a：Bard 時代補足

※ 公開誌面では出さず、裏台帳だけで使う運用も可能。

## 11. 記述フォーマット方針（Markdown / YAML）

### 11-1. 基本方針
制作の母艦は Markdown ベースとし、上部メタデータは YAML front matter で持つ構成を推奨する。

理由：
- 人間が読みやすい
- あとで変換しやすい
- メモや備考を書き足しやすい
- Git 管理や差分確認に向く

### 11-2. 推奨構成
上部：YAML front matter
下部：Markdown 本文

### 11-3. YAML に置く情報
- id
- title
- category
- subtype
- experience_level
- reader_level
- figure_type
- start_date
- end_date
- version_status
- pricing_note
- evaluation_date
- related_terms
- status

### 11-4. Markdown 本文に置く情報
- 一言定義
- 本文解説
- 実務での意味
- つまずきポイント
- 私のコメント
- コミュニティ補完メモ
- 出典メモ
- 備考

### 11-5. 理由
長文や補足や雑感まで全部 YAML に入れると読みにくくなる。
そのため、構造化しやすい短い属性だけを YAML に置き、説明やメモは Markdown 側に逃がす。

## 13. 読者レベル設計

### 13-1. 目的
読者によって必要な深さが違うため、各項目に「どのレベル向けか」を持たせる。

### 13-2. reader_level 案
- Level 1：言葉だけ知りたい
- Level 2：軽く触る・会話についていきたい
- Level 3：実際に手を動かしたい
- Level 4：開発を自走したい
- Level 5：原理や実装まで踏み込みたい
- Level 6：最先端・論文・アーキテクチャまで追いたい

### 13-3. 使い方
- 各ページに「この項目の推奨読者レベル」を持つ
- 文章内で「ここでは Level 2 までを扱う」などの表現に使える
- 勉強会や配布時の対象者調整にも使える

## 14. モデル・サービスの時変情報設計

### 14-1. 背景
モデルやサービスは、リリース時期や安全対策やマイナーアップデートによって印象・使い勝手・性能が変わることがある。
したがって「その感想や評価が、いつ時点のものか」を持たせる必要がある。

### 14-2. 追加カラム案
- evaluation_date
- observed_period
- release_phase
- change_note
- downgrade_or_upgrade_note
- rollback_note
- safety_change_note
- pricing_change_note

### 14-3. 使い方
- 「2026年4月時点の印象」と明記する
- 「2025年6月は良かったが、2025年8月には印象が変わった」などの差分を保持する
- 主観コメントを時系列に押しつぶさない

## 19. 序文・まえがきに書きたいこと（箇条書き整理）

### 12-1. 原体験
- 自分にとって当たり前の言葉が、相手には通じない場面が増えた
- その断絶に驚いた
- 話している日本語は同じでも、同じ言語空間にいない感覚があった
- コンテキストやディクテーションのような言葉すら共有されていなかった

### 12-2. 勉強会・対話で感じたこと
- 内容をかなり絞っても、それでも難しいと言われた
- 自分では相当レベルを落としているつもりだった
- しかし前提知識の差が大きすぎて、説明が成立しない場面があった
- 自分だけで会話のレベル調整をすることに限界を感じた
- 企画側・橋渡し役の人を立てないと話が通らないと感じた

### 12-3. もどかしさ
- ハウツーだけにすると、感動や思想や原体験が抜け落ちる
- しかし原理原則から全部話すと長すぎて終わらない
- 「なぜそこにたどり着いたか」「何に心を動かされたか」が共有されない
- 触っている人にしか見えない実感が伝えづらい

### 12-4. 比喩として使えそうなもの
- 電気のない時代の人にコンピュータを説明するような難しさ
- そもそも前提となる概念が存在しないため、説明の足場がない
- 真空管やAND回路以前の層で止まってしまうような感覚

### 12-5. この本を作る理由
- 毎回その場で説明するのでは限界がある
- 共有語彙の辞書が必要だと感じた
- 自分の言っていることの意味を、辞書引きのように参照できる形にしたい
- 会話や仕事を成立させるための共通土台が必要だった
- 自分の学びの体系化でもあり、他者との橋渡しのための本でもある

### 12-6. 読者への位置づけ
- これは完全な教科書ではない
- これは現場で触りながら学んだ言葉の地図である
- 実体験と未体験を分けて書く
- わからないところや補うべきところは、コミュニティや実践で更新していく前提である
